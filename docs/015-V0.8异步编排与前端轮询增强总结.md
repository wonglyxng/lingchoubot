## 关键实现说明

### 异步工作流执行

- `Engine.RunAsync()` 方法：先调用 `workflow.CreateRun()` 创建运行记录，然后启动 goroutine 异步执行
- goroutine 使用 `context.Background()` 脱离 HTTP 请求生命周期，避免连接关闭导致工作流中断
- Handler 返回 `202 Accepted` 而非 `201 Created`，语义更准确
- 原同步 `Run()` 方法保留但标注建议使用 `RunAsync`

### 前端轮询机制

- `RunCard` 组件检测到 `running`/`pending` 状态时，自动每 2 秒轮询 `GET /api/v1/orchestrator/runs/{id}`
- 轮询使用 `useRef` + `setInterval`，在组件卸载或运行完成时自动清理
- 运行中的卡片显示旋转的 `RefreshCw` 图标
- 新建运行自动插入列表顶部并展开，无需刷新整个列表

### 死代码清理

- `RunStore`（内存 Map 存储）、`WorkflowRun`/`Step` 类型、状态枚举、`newID()` 函数在 WP2-01 DB 持久化后已完全废弃
- 通过 grep 确认零外部引用后安全移除

## 已知限制

1. 异步执行的错误仅记录到日志和数据库，没有推送通知机制（需要 WebSocket/SSE）
2. 轮询间隔固定 2 秒，未做指数退避
3. 如果服务重启，正在执行的 goroutine 会丢失（需要持久化任务队列解决）
4. 前端轮询在页面切走后仍会继续直到组件卸载
