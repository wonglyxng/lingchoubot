# 051-统一评审评分卡与返工治理设计

## 1. 文档信息

| 字段 | 值 |
|------|------|
| 文档名称 | 统一评审评分卡与返工治理设计 |
| 文档版本 | V0.1 |
| 文档状态 | Draft |
| 适用范围 | 全任务类型评审闭环 |
| 关联文档 | `001-灵筹系统总体架构设计`、`004-数据库设计`、`005-API 接口文档`、`021-WP2-07组织角色矩阵与主管路由设计` |

---

## 2. 背景与问题

当前评审闭环已经具备以下能力：

- Worker 产出工件后进入 `in_review`
- Reviewer 给出 `approved` 或 `needs_revision`
- `needs_revision` 会把任务打回 `revision_required`
- 返工最多允许 3 次

但当前实现仍存在三个核心问题：

1. **评审依据不够结构化**  
   Reviewer 主要依据通用 prompt 文字规则做判断，缺少任务级、可复盘的评分口径。

2. **通过标准不够显式**  
   当前更接近“内容充分则通过”，但没有把“充分”的标准落成固定模板、分项评分和阈值。

3. **返工动作不够可执行**  
   `needs_revision` 后虽然有状态回退，但没有把“哪些项没过、必须改什么、主管如何重新安排”结构化表达清楚。

这会导致需求梳理、PRD、架构设计等文档型任务容易在“执行 -> 评审 -> 修订”之间循环，但参与者无法明确回答：

- 为什么被打回
- 打回依据是什么
- 通过标准是什么
- 本轮返工到底要改什么

---

## 3. 设计目标

本方案目标如下：

1. 为所有任务建立统一评审框架，但允许按任务类型使用不同评分模板。
2. 将“评审结论”从自然语言判断升级为“硬门槛 + 加权评分”的结构化结果。
3. 让 `approved` 与 `needs_revision` 都有明确依据，可用于审计、审批、前端展示与返工安排。
4. 让 `needs_revision` 后的下一步动作回到责任主管，而不是只做原地重跑。
5. 保持 MVP 阶段可实现，不引入过度抽象。

---

## 4. 非目标

当前方案不覆盖以下事项：

1. 不引入完全自由配置的任意评分 DSL。
2. 不把评审结果直接替代人工审批。
3. 不试图在第一版就为每个细分任务族定义高度复杂的行业级评分规范。
4. 不推翻现有 `review_report.verdict` 模型；第一版默认 LLM Reviewer 仍使用 `approved / needs_revision`，`rejected` 保留给人工评审或兼容性场景。

---

## 5. 核心设计原则

### 5.1 先过硬门槛，再看总分

总分不能掩盖关键缺陷。任何关键项未满足时，必须直接 `needs_revision`。

### 5.2 模板优先于自由发挥

评分标准必须优先来自任务类型模板，而不是 Reviewer 临场自定义。

### 5.3 有限覆盖，不允许任意漂移

主管可以在任务契约中对模板做小幅调整，但不能推翻统一口径。

### 5.4 返工必须有明确修订指令

`needs_revision` 必须附带具体未通过项、失分项和必改项，不能只输出“请补充”。

### 5.5 评审结果必须可展示、可审计、可继续执行

结构化评分结果不仅用于判断通过与否，还要能直接支撑：

- 审计日志
- 审批依据
- 控制台展示
- 主管生成返工 brief

---

## 6. 总体方案

引入“默认评分模板 + 任务契约覆盖 + 结构化评审结果”的统一评审机制。

执行顺序如下：

1. Supervisor 创建任务契约时，绑定一个默认评分模板。
2. 如有必要，Supervisor 在允许范围内做有限覆盖。
3. Worker 执行任务并产出工件。
4. Reviewer 读取任务、完整契约、工件和评分模板。
5. Reviewer 逐项输出：
   - 硬门槛结果
   - 分项得分
   - 总分
   - verdict
   - 必改项
   - 建议项
6. 服务层依据固定规则决定是否推进：
   - 硬门槛失败 -> `needs_revision`
   - 硬门槛全过但总分低于阈值 -> `needs_revision`
   - 硬门槛全过且总分达到阈值 -> `approved`
7. 若 `needs_revision`，任务进入 `revision_required`，并回到责任主管生成返工 brief，再重新安排执行。

---

## 7. 任务类别与默认模板

第一版建议覆盖六类任务：

| task_category | 适用任务 | 默认模板 |
|------|------|------|
| `prd` | 需求梳理、PRD、范围澄清 | `prd_v1` |
| `architecture` | 技术可行性评估、架构设计、方案设计 | `architecture_v1` |
| `backend` | 后端开发、API、数据库、服务实现 | `backend_v1` |
| `frontend` | 页面、组件、交互实现 | `frontend_v1` |
| `qa` | 测试计划、测试执行、回归验证 | `qa_v1` |
| `release` | 发布计划、部署、回滚、运维交付 | `release_v1` |

说明：

- `task_category` 由 Supervisor 根据任务目标显式确定，避免仅依赖标题关键词猜测。
- Worker 专业化路由应优先复用契约中的 `task_category` 映射结果，保证执行者选择、评审模板与返工链路使用同一分类来源。
- 当请求的 worker specialization 没有专属活跃 Agent 时，应回退到 `general` worker，而不是在编排层直接失败。
- 关键词兜底时，`API / 接口 / 数据库 / 服务实现` 归入 `backend`；`architecture` 仅用于可行性评估、架构方案、技术方案类任务。
- 关键词兜底的匹配顺序应以任务标题为第一优先级，任务描述仅作为补充证据，避免描述中的泛化词汇（如“文章发布”“权限验证”）反向污染任务类别。
- 执行域推断中，`backend / frontend / architecture / release` 均归入 `development` 主管域，`qa` 归入 `qa` 主管域，`prd` 维持 `general`。
- 推断逻辑可作为兜底，但不能成为唯一依据。
- PM 输出在落库前应做项目目标绑定校验；校验粒度应基于 `summary + phase + task` 的联合上下文，避免 generic 任务标题因未重复项目名而被误判，同时要在明显跨业务域漂移时立即阻断链路。
- 如果项目描述中存在显式非目标约束，例如“仅前端”“不包含后端 API / CMS / 后台管理”，PM 输出还应额外通过范围越界校验；允许在 summary/phase 中复述这些非目标，但不能把它们拆成真实执行任务。

## 8. 评分模板结构

每个模板采用统一结构：

```json
{
  "template_key": "prd_v1",
  "task_category": "prd",
  "pass_threshold": 80,
  "hard_gates": [
    {
      "key": "scope_defined",
      "name": "范围明确",
      "description": "必须明确本轮范围与边界"
    }
  ],
  "score_items": [
    {
      "key": "completeness",
      "name": "完整性",
      "weight": 25,
      "description": "关键部分是否完整覆盖"
    }
  ],
  "override_policy": {
    "threshold_min": 75,
    "threshold_max": 90,
    "max_extra_items": 2,
    "allow_weight_adjustment": true,
    "allow_remove_global_hard_gates": false
  }
}
```

### 8.1 全局硬门槛

所有模板都必须带以下全局硬门槛：

1. `goal_match`  
   工件与任务目标一致，不能答非所问。

2. `no_placeholder`  
   不能存在模板占位、待补充、空洞泛话。

3. `project_binding`  
   必须明确绑定当前项目与当前任务，而不是通用产物。

### 8.2 任务类型专属硬门槛与评分项

#### `prd_v1`

硬门槛：

- 目标与范围明确
- 非目标明确
- 核心流程或核心场景齐全
- 验收标准可验证
- 风险与未决问题已列出

评分项：

- 完整性 `25`
- 边界清晰度 `20`
- 内部一致性 `15`
- 可执行性 `20`
- 验收可测性 `20`

#### `architecture_v1`

硬门槛：

- 问题背景明确
- 方案结论明确
- 依赖与资源评估明确
- 主要风险明确
- 推荐方案及取舍明确

评分项：

- 技术可行性 `25`
- 取舍说明 `20`
- 约束一致性 `15`
- 实施指导性 `20`
- 风险控制 `20`

#### `backend_v1`

硬门槛：

- 实现覆盖任务范围
- 无明显占位代码
- 关键错误处理存在
- 接口与数据结构与契约一致
- 若契约要求测试，则必须有测试证据

评分项：

- 功能正确性 `35`
- 契约符合度 `20`
- 健壮性 `15`
- 代码可读性 `15`
- 测试与验证 `15`

#### `frontend_v1`

硬门槛：

- 页面或组件满足任务目标
- `loading / empty / error` 状态完整
- 真实 API 对接符合要求
- 无明显占位 UI
- 基础响应式与可访问性满足最低要求

评分项：

- 需求符合度 `25`
- 交互完整性 `20`
- 状态处理 `20`
- 代码质量 `15`
- 可用性 `10`
- 可访问性 `10`

#### `qa_v1`

硬门槛：

- 测试范围明确
- 执行步骤可复现
- 结果有证据
- 缺陷与风险有记录
- 结论明确

评分项：

- 覆盖度 `30`
- 复现性 `20`
- 缺陷质量 `20`
- 风险针对性 `15`
- 自动化价值 `15`

#### `release_v1`

硬门槛：

- 发布步骤明确
- 回滚方案明确
- 配置与依赖变更明确
- 验证步骤明确
- 风险与监控方案明确

评分项：

- 安全性 `30`
- 完整性 `20`
- 可操作性 `20`
- 可观测性 `15`
- 表达清晰度 `15`

## 9. 有限覆盖规则

为平衡可控性与灵活性，Supervisor 允许在任务契约中做有限覆盖：

1. 通过阈值允许调整到 `75-90`。
2. 单项权重允许小幅调整，但总和必须保持 `100`。
3. 最多新增 `2` 个专项评分项；若 Supervisor 输出超过 `2` 个新增评分项，系统按 `weight` 降序、`key` 升序保留前 `2` 个，并将被裁剪项写入留痕元数据。
4. 不允许删除全局硬门槛。
5. 专属硬门槛原则上只能新增，不能删除。
6. 若使用模板覆盖，Supervisor 必须同时提供：
   - `review_policy_reason`：说明为什么默认模板不足、专项要求是什么
   - `review_policy_source`：说明专项要求依据来自哪里，例如 `task.description`、`acceptance_criteria`、`user instruction`

建议在 `task_contract.metadata.review_policy` 或等价结构中保存覆盖内容，并额外保存 `review_policy_override_reason`、`review_policy_override_source` 等留痕字段。

## 10. Reviewer 输入要求

Reviewer 必须拿到完整评审上下文，至少包括：

1. 任务基础信息
2. 完整任务契约
3. 评分模板
4. 模板覆盖结果
5. 当前工件与版本内容
6. 历史评审摘要与返工次数

其中，任务契约至少应包含：

- `scope`
- `done_definition`
- `verification_plan`
- `acceptance_criteria`

若 Reviewer 未拿到完整契约与模板，则评审结果应视为上下文不完整。MVP 第一版可先按 `failed / waiting_manual_intervention` 路径处理，而不是强行给出正式 verdict。

---

## 11. Reviewer 输出结构

每次任务评审仍然只允许输出 1 条任务级结论，但内容升级为结构化评分结果：

```json
{
  "status": "success",
  "summary": "PRD 结构完整，但验收标准可测性不足，需修订",
  "reviews": [
    {
      "template_key": "prd_v1",
      "pass_threshold": 80,
      "total_score": 74,
      "verdict": "needs_revision",
      "hard_gate_results": [
        {
          "key": "acceptance_testable",
          "passed": false,
          "reason": "验收标准只有原则描述，没有可验证口径"
        }
      ],
      "score_items": [
        {
          "key": "completeness",
          "weight": 25,
          "score": 22,
          "max_score": 25,
          "reason": "主体章节基本完整"
        }
      ],
      "must_fix_items": [
        "补充可逐条验证的验收标准",
        "补充异常场景与边界约束"
      ],
      "suggestions": [
        "可增加角色权限矩阵提升可执行性"
      ]
    }
  ]
}
```

### 11.1 输出约束

1. 只能输出 1 条任务级 review。
2. `verdict` 只能是 `approved` 或 `needs_revision`。
3. `hard_gate_results` 必须覆盖本次启用的全部硬门槛。
4. `score_items` 必须覆盖本次启用的全部评分项。
5. `needs_revision` 时，`must_fix_items` 至少 1 条。
6. 评分项的 `score` 不能大于 `max_score`。

---

## 12. 通过与打回规则

服务层必须按固定规则解释评审结果，而不是仅信任 Reviewer 的自然语言摘要。

判定逻辑如下：

1. 任一硬门槛失败 -> `needs_revision`
2. 硬门槛全通过，但 `total_score < pass_threshold` -> `needs_revision`
3. 硬门槛全通过，且 `total_score >= pass_threshold` -> `approved`

说明：

- `80` 分是默认阈值，不是唯一标准。
- Reviewer 不得通过“高分”掩盖硬门槛失败。
- 总分仅在硬门槛全部满足后才有意义。

## 13. 返工闭环设计

### 13.1 返工原则

`needs_revision` 后，不应直接原地再次执行 Worker，而应回到责任主管链路。

### 13.2 闭环流程

```text
Worker 产出工件
 -> Reviewer 结构化评分
 -> 若 needs_revision:
      1. 写入 review_report
      2. 任务状态转为 revision_required
      3. 生成 rework_brief
      4. 回到责任主管
      5. 主管决定继续执行者 / 更换执行者 / 升级人工澄清 / 拆分任务
      6. 重新进入执行
```

### 13.3 rework_brief 内容

主管接手返工时，应看到结构化返工摘要，至少包含：

- 第几次返工
- 未通过的硬门槛
- 主要失分项
- 本轮必须修复项
- 建议优化项
- 是否需要额外输入或人工澄清

MVP 第一版不必单独建表，可先将 `rework_brief` 写入：

- `review_report.metadata`
- 审计事件 `task.rework`
- 或任务 `metadata.current_rework_brief`

### 13.4 返工上限

保留当前“最多 3 次返工”的保护阈值，但其意义改为：

- 系统允许最多 3 次受控返工
- 每次返工都必须有明确 `rework_brief`
- 超限后不应只报错结束，而应进入 `waiting_manual_intervention`
- `waiting_manual_intervention` 至少要区分两类原因：
  - `llm_execution_failed`
  - `rework_limit_reached`
- 前端文案和可执行动作必须跟随原因类型变化，不能把返工超限误展示成“供应商/模型故障”

### 13.5 人工介入最小闭环

MVP 第一版不做复杂在线编辑器，但人工介入至少要支持一个真正可推进链路的动作：

- 对 `rework_limit_reached` 场景，允许“人工放行到审批”
- 该动作不是直接把任务标记为 `completed`
- 正确状态流应为：
  1. 人工确认当前交付可接受
  2. 任务从 `revision_required` 显式转入 `pending_approval`
  3. 自动创建一条审批请求
  4. 工作流从 `waiting_manual_intervention` 恢复，并自然进入 `waiting_approval`

建议将人工介入上下文写入 `workflow_run.metadata.manual_intervention`，至少包含：

- `reason_code`
- `reason`
- `agent_role`
- `phase_id`
- `task_id`
- `task_title`
- `available_actions[]`

## 14. 数据模型建议

### 14.1 task_contract

`task_contract` 已有 `metadata` 字段。MVP 第一版建议优先写入 `metadata.review_policy`，避免一开始就扩表：

- `task_category`
- `review_template_key`
- `review_policy`
  - `pass_threshold`
  - `hard_gates`
  - `score_items`
  - `override_metadata`

若后续查询和索引需求明显增大，再考虑把 `task_category`、`review_template_key` 升格为显式列。

### 14.2 review_report

建议在 `metadata` 中结构化保存：

- `template_key`
- `task_category`
- `pass_threshold`
- `total_score`
- `hard_gate_results[]`
- `score_items[]`

### 14.3 workflow_run

建议为 `workflow_run` 增加 `metadata`，用于承载运行态半结构化上下文。当前优先写入：

- `manual_intervention`
  - `reason_code`
  - `reason`
  - `agent_role`
  - `phase_id`
  - `task_id`
  - `task_title`
  - `available_actions[]`

这样工作流页才能根据真实阻塞原因展示不同说明与人工动作，而不是统一靠 `error` 文本猜测。
- `must_fix_items[]`
- `suggestions[]`
- `rework_count`

### 14.3 approval_request

若评审通过自动创建审批，则建议在 `approval_request.metadata` 中继续带出：

- `template_key`
- `total_score`
- `hard_gate_passed_count`
- `score_breakdown_summary`
- `review_summary`

这样审批方看到的不只是“已通过”，还能看到“按什么模板、多少分通过”。

## 15. API 与前端展示建议

### 15.1 API

后续建议扩展：

- `POST /api/v1/reviews`
  - 接收完整结构化评分结果
- `GET /api/v1/reviews/{id}`
  - 返回分项评分、硬门槛结果、必改项
- `GET /api/v1/tasks/{id}`
  - 可附带当前任务绑定的评分模板摘要

### 15.2 前端

评审页与审批页建议直接展示：

- 总分：如 `74 / 100`
- 通过阈值：如 `80`
- 硬门槛通过情况：如 `4 / 5`
- 未通过硬门槛列表
- 主要失分项
- 必改项
- 当前返工次数

这样用户和主管都能直接看懂为什么被打回或为什么通过。

## 16. 落地顺序建议

建议分三步落地，避免一次改太大：

### Step 1：先补评审输入与输出结构

- Reviewer 输入补齐完整契约与评分模板
- Reviewer 输出扩展为结构化评分结果
- 仍保持现有 verdict 驱动流转

### Step 2：落模板与服务层判定

- 引入默认模板注册表
- 实现模板覆盖规则
- 服务层按硬门槛和阈值解释 review 结果

### Step 3：接主管返工闭环

- `needs_revision` 后不再直接原地重跑 Worker
- 生成 `rework_brief`
- 让责任主管重新安排执行

---

## 17. 验收标准

本方案落地后，应满足以下验收标准：

1. 任一任务评审都能明确看到使用的模板、阈值、分项评分和硬门槛结果。
2. Reviewer 不再只返回自然语言结论，而是返回结构化评分结果。
3. `needs_revision` 时，必须能看到明确的 `must_fix_items`。
4. `approved` 时，审批方可直接看到通过分数与评分依据。
5. `revision_required` 后任务回到责任主管链路，而不是仅原地重跑 Worker。
6. 至少覆盖 `prd`、`architecture`、`backend`、`frontend`、`qa`、`release` 六类任务模板。

---

## 18. 已知取舍

1. 模板化评分会增加 prompt 和返回结构复杂度，但换来更稳定的口径与更强的可审计性。
2. 第一版模板不可避免会偏粗，需要在真实运行中不断修订。
3. 对于非常小的任务，完整评分卡可能显得偏重，因此后续可考虑提供“轻量模板”，但仍应保持硬门槛机制。

---

## 19. 结论

“超过 80 分就通过”可以作为统一评审机制的一部分，但不能作为唯一标准。

灵筹更合适的做法是：

- 用任务类型模板定义评分项
- 用硬门槛保证关键质量底线
- 用加权总分控制整体通过阈值
- 用结构化评审结果指导返工与审批

这样才能真正回答：

- 为什么打回
- 打回依据是什么
- 通过标准是什么
- 下一轮该怎么改

这比单纯“Reviewer 主观判断是否充分”更符合灵筹“结构、秩序、控制、审计”的工程目标。
