# 043-SSE事件语义标准化与控制台增量刷新总结

## 变更摘要

本次继续收口控制台实时刷新链路，完成两项工作：

1. 后端将 `workflow` 主题 SSE 事件的顶层 `target_id` 标准化为 `workflow_run.id`
2. 前端 `audit`、`tool-calls` 页面改为增量更新或静默刷新，不再沿用容易造成闪烁的全量重载方式

## 关键实现说明

### 1. workflow SSE 顶层标识语义收口

此前 `workflow` 事件的顶层 `target_id` 实际上传的是 `project_id`，前端必须从 `data.after_state.run_id` 再解析一次运行 ID。

本次在 `AuditService.publishEvent()` 中增加了 `workflow` 事件专用标准化逻辑：

- 顶层 `target_id` 统一设置为 `workflow_run.id`
- 顶层 `project_id` 单独保留所属项目 ID
- `data` 中的原始审计日志对象保持不变，继续保留审计语义

这样前端消费 workflow 实时事件时，可以直接使用顶层 `target_id` 定位运行对象。

### 2. audit 页面改为事件增量写入

- 收到实时事件后，直接从 `event.data` 解析审计日志对象
- 将新日志合并到当前列表并按时间倒序整理
- 仅在事件解析失败或 SSE 降级轮询时，才执行静默列表刷新

### 3. tool-calls 页面改为单条刷新

- 收到 `tool_call` 事件后，优先使用顶层 `target_id` 调 `GET /api/v1/tool-calls/{id}`
- 返回成功后只合并更新单条记录
- 仅在单条查询失败或 SSE 降级时，才静默重拉列表

### 4. workflow 页面同步去除 run_id 解析绕路

- workflow 页优先直接使用事件顶层 `target_id`
- 仍保留对旧 `data.after_state.run_id` 的兼容回退，便于平滑升级
