# WP2-10 CI级端到端集成测试实现总结

## 变更摘要

实现了 WP2-10 的核心目标：基于纯内存 fake 仓库的 Engine 端到端集成测试，覆盖 Happy Path、返工路径、最大返工上限、审计日志全面性、工件版本递增共 5 个场景。

关键基础设施：
1. **Service 层接口提取** — 将 13 个 service 的 repo 依赖从具体类型改为导出接口，支持测试注入
2. **testutil 包** — 提供 15 个内存 fake repo、Fixture 构建器、标准 Agent 种子数据
3. **5 个集成测试** — 覆盖引擎核心链路的正确性、返工循环、边界条件

## 关键实现说明

### Service 层接口提取
- `repo_interfaces.go` 定义了 16 个导出接口（ProjectRepository、PhaseRepository 等）
- 每个接口精确匹配对应 `*repository.XxxRepo` 的方法签名
- 13 个 service 的结构体字段和构造函数参数全部从 `*repository.XxxRepo` 改为导出接口
- 对于已有内部接口的 4 个 service（task、approval、audit、workflow），移除旧接口改用统一导出接口

### testutil 包
- 15 个 fake repo 使用 `map[string]*model.Xxx` + `sync.Mutex` 实现
- `Fixture` 结构体持有所有 fake repo、13 个 service、Engine、Registry
- `NewFixture()` 完成全量依赖注入，注册默认 mock agents
- `SeedStandardAgents()` 创建 7 个标准 agent（PM、Dev/QA Supervisor、3 个 Worker、Reviewer）

### 集成测试场景
| 测试名 | 场景 | 关键断言 |
|--------|------|----------|
| `TestIntegration_HappyPath` | 全链路正常执行 | 4 阶段、9 任务、9 工件+版本、9 评审、28 步 |
| `TestIntegration_ReworkPath` | 每任务返工一次 | 18 评审、18 工件、9 rework 审计事件、46 步 |
| `TestIntegration_MaxReworkExceeded` | 始终打回触发上限 | 36 评审（9×4 轮） |
| `TestIntegration_AuditLogCoverage` | 审计日志事件类型全面性 | 12 种必须事件类型全部出现 |
| `TestIntegration_ArtifactVersioning` | 返工后工件版本递增 | 18 工件、18 版本 |

### 自定义 Mock Agent
- `reworkReviewer`：按 taskID 追踪调用次数，首次评审打回，后续通过
- `alwaysRejectReviewer`：始终返回 needs_revision，用于测试最大返工上限

## 运行方式

```bash
cd backend

# 运行集成测试
go test -v -count=1 ./internal/orchestrator/ -run TestIntegration

# 运行全量测试
go test -count=1 ./...
```

## 测试方式

所有测试均为纯内存级别，无需数据库或外部依赖：

```bash
# 全量测试（约 5 秒）
cd backend && go test -count=1 ./...

# 仅集成测试
go test -v ./internal/orchestrator/ -run TestIntegration
```

## 已知限制

1. **Temporal 模式集成测试** — 使用 Temporal SDK testsuite 的内存环境，不测试真实 Temporal Server 连接
2. **MinIO 存储路径** — fake ArtifactRepo 不模拟真实文件存储
3. **并发安全** — fake repo 使用 Mutex 保护，但仅适用于单测场景
4. **Task 最终状态** — Happy Path 中任务停留在 `in_review`，未自动转为 `completed`（与引擎设计一致，completion 由外部触发）
5. **前端 CI** — 需要 `package-lock.json` 存在才能使用 `npm ci`，否则 CI 前端 job 会跳过
