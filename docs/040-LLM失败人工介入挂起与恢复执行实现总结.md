# 040-LLM失败人工介入挂起与恢复执行实现总结

## 1. 变更摘要

本次为工作流主链路补上了“LLM 失败后上报并等待人工介入”的显式状态机，而不是直接把整个运行打成 `failed`。

具体效果如下：

1. 当 PM、Supervisor、Worker、Reviewer 的真实 LLM 调用返回失败时，`workflow_run.status` 进入 `waiting_manual_intervention`。
2. 错误原因会保留在 `workflow_run.error`，同时写入 `workflow.waiting_manual_intervention` 审计事件。
3. 新增 `POST /api/v1/orchestrator/runs/{id}/resume` 接口，用于人工处理后恢复执行。
4. 恢复时不再一律从头跑完整链路，而是按当前任务状态从 PM、Worker、Reviewer 的合适阶段继续。
5. Temporal 分支同步支持该语义：LLM 失败时不再把 run 覆写为 `failed`，而是停在等待人工介入状态。

## 3. 关键实现说明

### 3.1 新增运行态状态：`waiting_manual_intervention`

在 `workflow_run.status` 中新增 `waiting_manual_intervention`，用于表达“真实 LLM 调用失败，但流程未终止，等待人工处理后恢复”。

这与 `waiting_approval` 的区别是：

- `waiting_approval` 是业务正常推进后的审批关口暂停。
- `waiting_manual_intervention` 是执行异常后的人工处理暂停。

### 3.2 本地引擎把 LLM 失败识别为可恢复中断

在本地 `Engine` 中，PM / Supervisor / Worker / Reviewer 如果返回 `OutputStatusFailed`，不再直接走 `workflow.failed`。

改为：

1. 先把当前 `workflow_step` 标记为 `failed`，保留这次失败尝试的上下文。
2. 再把 `workflow_run` 标记为 `waiting_manual_intervention`。
3. 记录 `workflow.waiting_manual_intervention` 审计事件，包含 `run_id`、`agent_role`、`task_id`、`phase_id`、`error` 等信息。

真正的非 LLM 失败，例如数据库写入、对象存储、流程内部错误，仍然沿用 `workflow.failed` 语义。

### 3.3 恢复执行改为按当前任务状态继续

恢复后如果一律从 Supervisor 重新开始，会导致重复创建契约、分派或工件。因此本次把任务链入口改造成状态感知：

- `pending`：从 Supervisor 开始。
- `assigned` / `in_progress` / `revision_required`：从 Worker 开始。
- `in_review`：直接从 Reviewer 开始。
- `completed` / `pending_approval`：跳过，不重复执行。

这使得 Worker/Reviewer 阶段发生 LLM 失败后，人工处理完再恢复时，可以尽量从上次失败点继续，而不是重跑上游成功步骤。

### 3.4 PM 失败恢复特殊处理

如果失败发生在 PM 分解阶段，恢复时需要重新进入 PM，而不能直接执行 `continueRun`。因此增加了对“最后一个失败步骤是否为 PM”的判定逻辑，确保 PM 失败后恢复会先重新分解项目。

### 3.5 API 层新增 resume 接口

新增：

- `POST /api/v1/orchestrator/runs/{id}/resume`

该接口允许恢复两类暂停运行：

- `waiting_approval`
- `waiting_manual_intervention`

### 3.6 Temporal 分支同步支持人工介入挂起

Temporal activities 在遇到 LLM 失败时，会先把数据库中的 `workflow_run` 置为 `waiting_manual_intervention`，然后返回不可重试错误；顶层 Temporal workflow 识别到该类错误后直接结束，不再调用 `ActivityFailRun` 把状态覆写回 `failed`。

本次同时让 `WorkflowEngine` 统一暴露 `ResumeRun`，并在 API 启动时把恢复器统一挂到 `approvalSvc` 上，使本地/Temporal 两种部署形态都能走同一恢复入口。

## 4. 运行方式

1. 执行 migration：

```bash
cd backend
go run ./cmd/migrate
```

或使用 compose：

```bash
docker compose up --build -d migrate api
```

2. 启动 API。

3. 发起工作流：

```http
POST /api/v1/orchestrator/runs
```

4. 若真实 LLM 调用失败，轮询 `GET /api/v1/orchestrator/runs/{id}`，应看到：

- `status = waiting_manual_intervention`
- `error` 中包含最近失败原因

5. 人工修复后调用：

```http
POST /api/v1/orchestrator/runs/{id}/resume
```

## 6. 已知限制

1. Temporal 分支的恢复实现当前复用了本地 `Engine.ResumeRun` 的继续执行逻辑，语义已统一，但恢复后的继续执行不再由原 Temporal workflow 实例承接。
2. 本次仅对“真实 LLM 调用失败”做人工介入挂起；数据库、对象存储、流程内部一致性等非 LLM 错误仍会直接落成 `failed`。
3. 本次没有修改前端页面来专门高亮 `waiting_manual_intervention`，控制台若要提供更友好的人工介入操作入口，还需要前端后续补充。
