# 028-评审审批联动与待审批状态实现总结

## 变更摘要

修复评审通过后任务状态不推进、审批中心为空的流程断裂问题。新增 `pending_approval`（待审批）任务状态，打通评审→审批→完成的完整闭环。

核心改动：
1. 任务状态机新增 `pending_approval`，位于 `in_review` 和 `completed` 之间
2. 评审报告 verdict=approved 时，自动将任务推进到 `pending_approval` 并创建审批请求
3. 审批通过 → completed，审批拒绝 → revision_required
4. 前端任务看板新增"待审批"列，状态映射新增对应展示

## 关键实现说明

### 状态流转变化

```
旧: in_progress → in_review → completed (评审通过时无推进)
新: in_progress → in_review → pending_approval → completed
                                  ↘ revision_required (审批拒绝)
                    ↘ revision_required (评审拒绝/需修订)
```

### 评审→审批联动

`ReviewReportService.Create` 在 verdict=approved 时：
1. 调用 `TaskService.TransitionStatus(taskID, pending_approval)`
2. 查询任务信息获取 project_id
3. 调用 `ApprovalRequestService.Create` 自动创建审批单

### 依赖注入

采用 `SetApprovalService` 后注入方式，避免 ReviewReportService 和 ApprovalRequestService 之间的循环初始化依赖。

## 已知限制

1. MVP 阶段所有评审通过的任务都会自动进入待审批，暂无"跳过审批直接完成"的开关
2. 已有数据库中处于 `in_review` 状态且评审已通过的任务不会自动迁移，需手动处理或重新触发评审
3. 前端审批中心暂无"发起审批"按钮（依赖自动创建），后续可按需补充手动发起入口
