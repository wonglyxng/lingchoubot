# 050-审批决策告警语义与恢复失败收口总结

## 1. 变更摘要

本次修复了审批接口的一个次生问题：审批请求在数据库中已经成功决策、关联任务状态也已更新，但因为后续恢复工作流运行失败，接口仍向前端返回 `400 DECIDE_FAILED`，造成“页面提示失败、数据其实已改成功”的误导。

修复后：

1. 审批决策成功即返回 `200`。
2. 工作流恢复、阶段上下文读取、`run_id` 解析等后续动作改为“成功后的继续动作”。
3. 若继续动作失败，响应中返回告警字段，并写入审计事件，不再把整个审批决策打成失败。

## 3. 关键实现说明

### 3.1 决策结果显式建模

新增 `ApprovalDecisionResult`，用于表达：

- 审批状态是否已写成功
- 关联任务状态是否已回写
- 关联工作流运行是否已恢复
- 是否存在继续动作告警

这样前端和调用方可以明确区分“审批失败”和“审批成功但后续恢复未完成”。

### 3.2 恢复失败改为告警而非接口失败

`ApprovalRequestService.Decide` 现在将流程拆成两段：

1. 审批状态落库 + 审计事件 + 任务状态回写
2. 阶段收口后的工作流恢复尝试

第一段一旦成功，就返回成功响应。第二段若失败，不回滚审批结果，而是在 `warnings[]` 中返回告警，并写入 `approval_request.continuation_warning` 审计事件。

### 3.3 请求字段兼容

审批决策接口现在同时兼容：

- `note`
- `decision_note`

避免前后端字段名不一致时 silently drop 决策备注。

## 4. 运行方式

1. 启动系统：`docker compose up --build`
2. 创建项目并运行工作流至 `waiting_approval`
3. 对审批请求调用 `POST /api/v1/approvals/{id}/decide`
4. 如果审批成功但工作流运行无法恢复，接口应返回 `200`，并在响应中带上 `workflow_resume_status="warning"`

## 5. 测试方式

建议执行：

1. `cd backend && go test ./internal/service -run ApprovalRequestServiceDecide -count=1`
2. 在真实接口上重放一次“run 已 failed 后再审批”的场景，确认响应为 `200` 且包含告警字段

## 6. 已知限制

1. 当前仅把“决策成功后的继续动作失败”收敛为告警；并未把审批落库、任务回写、工作流恢复收拢为单数据库事务。
2. 若任务状态回写本身失败，当前实现同样返回成功并携带告警，这是为了避免再次出现“状态部分成功但接口整体失败”的不透明行为。
3. 前端目前只依赖请求成功与否，尚未把 `warnings[]` 直接渲染到审批页面；如需更强可观测性，可在后续 UI 中补充提示。
