## 2. 目标

本实施计划用于把 [统一评审评分卡与返工治理设计](C:/work/gospace/lingchoubot/docs/051-%E7%BB%9F%E4%B8%80%E8%AF%84%E5%AE%A1%E8%AF%84%E5%88%86%E5%8D%A1%E4%B8%8E%E8%BF%94%E5%B7%A5%E6%B2%BB%E7%90%86%E8%AE%BE%E8%AE%A1.md) 拆成可以逐步落地、逐步验证的工作包。

本计划要解决三件事：

1. 把评审从“自然语言主观判断”升级为“模板 + 硬门槛 + 加权评分”的结构化机制。
2. 让 `needs_revision` 有明确依据，并生成可执行的返工摘要。
3. 让返工重新回到责任主管链路，而不是只在 `worker -> reviewer` 之间原地循环。

---

## 3. 当前代码基线

结合当前仓库实现，评审闭环的现状如下：

1. `backend/internal/runtime/llm_prompts.go` 中 Reviewer prompt 仍以通用文字规则为主，只要求输出 `approved | needs_revision`、`findings`、`recommendations`。
2. `backend/internal/runtime/protocol.go` 中 `ContractCtx` 只定义了基础契约字段，但 orchestrator 实际传给 Worker 的契约仍只有 `scope`，传给 Reviewer 的上下文中甚至没有完整契约。
3. `backend/internal/runtime/llm_validator.go` 只覆盖最小结构校验和少量任务感知校验，尚未校验评分模板、硬门槛覆盖、总分阈值等规则。
4. `backend/internal/orchestrator/engine.go` 中返工逻辑虽然会记录 `task.rework` 审计事件，但任务进入 `revision_required` 后仍会继续走 `worker -> reviewer` 子链路，没有重新进入主管规划步骤。
5. `backend/internal/service/review_report.go` 会根据 verdict 推动任务状态和审批创建，但 `review_report.metadata` 尚未承载结构化评分结果。
6. 前端 `reviews`、`approvals` 页面目前只能展示摘要、发现项和建议项，无法直观看到评分、阈值、硬门槛与必改项。

结论：当前链路已经具备“执行 -> 评审 -> 审批/返工”的基本骨架，但还没有形成可解释、可审计、可指导返工的评审尺度。

## 4. 范围与约束

### 4.1 本计划范围

- 引入默认评分模板注册表
- 为任务契约绑定评审模板与覆盖策略
- 扩展 Reviewer 输入与输出结构
- 在 `review_report.metadata` 中落结构化评分结果
- 生成 `rework_brief`
- 将 `revision_required` 后的下一步动作重新路由回主管
- 前端补充评分卡和返工信息展示
- 同步更新数据库与 API 文档

### 4.2 不在本计划内

- 新增独立评分模板管理后台
- 增加复杂 DSL 式评分规则引擎
- 改造人工审批模型
- 新增独立 `rework_brief` 表
- 大范围重构现有前端页面结构

### 4.3 本轮实施约束

为控制首轮改造范围，本计划采用以下约束：

1. **首轮不加 migration**  
   优先复用现有 JSONB 字段，不先扩表。

2. **优先使用现有 metadata 字段**  
   复用：
   - `task_contract.metadata.review_policy`
   - `review_report.metadata`
   - `task.metadata.current_rework_brief`

3. **保持 verdict 兼容语义**  
   首轮默认 LLM Reviewer 仍使用 `approved | needs_revision`。现有 `review_report.verdict = rejected` 保留给人工评审或兼容场景，不在本轮推翻。

4. **按小步闭环推进**  
   每个工作包都必须能独立验证，不允许一次性同时大改模型、前端、文档、编排多个方向而无法回归。

---

## 5. 总体实施策略

建议按以下顺序推进：

1. 先建立评分模板与覆盖规则
2. 再扩展 Reviewer 的输入输出结构
3. 再把结构化结果写入服务层和审批上下文
4. 再改返工路由，让主管重新接手
5. 最后补前端展示和文档同步

这样做的好处是：

- 先把“尺度”定义清楚，再改工作流
- 先让数据结构落稳，再改页面
- 首轮不扩表，降低 schema 返工成本

## 7. 工作包拆解

### WP-01 评分模板与策略注册

**目标：** 为六类任务建立统一默认评分模板，并提供有限覆盖能力。

**新增/修改模块：**

- 新增 `backend/internal/reviewpolicy/types.go`
- 新增 `backend/internal/reviewpolicy/registry.go`
- 新增 `backend/internal/reviewpolicy/registry_test.go`

**执行内容：**

1. 定义评分模板、硬门槛、评分项、覆盖策略、已解析评审策略等核心类型。
2. 内置六类任务默认模板：
   - `prd`
   - `architecture`
   - `backend`
   - `frontend`
   - `qa`
   - `release`
3. 实现模板解析与覆盖逻辑，至少校验：
   - 阈值范围 `75-90`
   - 评分项权重总和为 `100`
   - 最多新增 `2` 个专项评分项
   - 不允许删除全局硬门槛
4. 输出 `ResolvedPolicy`，供 Supervisor、Reviewer、服务层统一使用。

**测试与验证：**

- `cd backend && go test ./internal/reviewpolicy -run TestResolvePolicy -v`
- `cd backend && go test ./internal/reviewpolicy -v`

**验收标准：**

- 默认模板可按任务类别正确解析
- 非法阈值和非法覆盖策略会被拒绝
- 解析结果可直接写入 `task_contract.metadata.review_policy`

**建议提交信息：**

```bash
git commit -m "feat: add review policy templates"
```

---

### WP-02 Reviewer 协议、Prompt 与校验器升级

**目标：** 让 Reviewer 从“摘要式评审”升级为“结构化评分卡输出”。

**新增/修改模块：**

- `backend/internal/runtime/protocol.go`
- `backend/internal/runtime/llm_prompts.go`
- `backend/internal/runtime/llm_validator.go`
- `backend/internal/runtime/llm_prompts_test.go`
- `backend/internal/runtime/llm_validator_test.go`

**执行内容：**

1. 扩展 `ContractCtx`，补齐：
   - `done_definition`
   - `verification_plan`
   - `acceptance_criteria`
   - `review_policy`
2. 扩展 `ReviewAction`，增加：
   - `template_key`
   - `pass_threshold`
   - `total_score`
   - `hard_gate_results`
   - `score_items`
   - `must_fix_items`
   - `suggestions`
3. 更新 Supervisor prompt，要求在契约输出中写入任务类别和评审策略。
4. 更新 Reviewer prompt，要求完整输出评分卡结构，而不是只给 `findings` / `recommendations`。
5. 更新校验器，至少校验：
   - 硬门槛结果是否全覆盖
   - 评分项结果是否全覆盖
   - 总分是否合法
   - `needs_revision` 是否包含 `must_fix_items`
   - `approved` 是否与硬门槛和阈值冲突

**测试与验证：**

- `cd backend && go test ./internal/runtime -run "Reviewer|Scorecard" -v`
- `cd backend && go test ./internal/runtime -v`

**验收标准：**

- Reviewer 未输出完整评分卡时会被校验器拦截
- Reviewer 无法在硬门槛失败时仍返回合法 `approved`
- `needs_revision` 没有 `must_fix_items` 时会校验失败

**建议提交信息：**

```bash
git commit -m "feat: add reviewer scorecard schema"
```

### WP-03 结构化评审结果落库与审批上下文增强

**目标：** 把结构化评分结果写入评审元数据，并带入审批上下文。

**新增/修改模块：**

- `backend/internal/orchestrator/engine.go`
- `backend/internal/service/review_report.go`
- `backend/internal/service/service_test.go`

**执行内容：**

1. 在 orchestrator 中补齐完整契约上下文的装配逻辑。
2. 在 Supervisor 生成或更新契约时，解析默认模板并将结果写入 `task_contract.metadata.review_policy`。
3. 在 Reviewer 执行前，把完整契约和评审策略传入输入上下文。
4. 在 `processReviewActions()` 中把以下内容合并写入 `review_report.metadata`：
   - `template_key`
   - `task_category`
   - `pass_threshold`
   - `total_score`
   - `hard_gate_results`
   - `score_items`
   - `must_fix_items`
   - `suggestions`
5. 在 `approvalMetadataFromReview()` 中继续补充评分摘要，让审批页能直接读取。

**测试与验证：**

- `cd backend && go test ./internal/service -run "Review|Approval" -v`
- `cd backend && go test ./internal/service ./internal/orchestrator -run "Review|Approval|Contract" -v`

**验收标准：**

- `review_report.metadata` 中可见完整评分卡结果
- 审批元数据能带出模板、总分和关键评审依据
- 不新增 schema 也能完成首轮闭环

**建议提交信息：**

```bash
git commit -m "feat: persist review scorecard metadata"
```

### WP-04 返工摘要与主管回收链路

**目标：** 让 `needs_revision` 后的下一步动作真正回到责任主管，而不是只重跑 Worker。

**新增/修改模块：**

- `backend/internal/orchestrator/engine.go`
- `backend/internal/service/review_report.go`
- `backend/internal/orchestrator/integration_test.go`
- `backend/internal/service/service_test.go`

**执行内容：**

1. 基于结构化评分结果生成 `rework_brief`，至少包含：
   - 第几次返工
   - 未通过的硬门槛
   - 主要失分项
   - 必改项
   - 建议项
   - 是否需要额外澄清
2. 先把 `rework_brief` 写入：
   - `review_report.metadata.rework_brief`
   - `task.metadata.current_rework_brief`
   - `task.rework` 审计事件 metadata
3. 修改 `revision_required` 后的路由：
   - 不再直接进入 `runWorker`
   - 重新进入 `runSupervisor`
   - 由责任主管读取 `rework_brief` 后重新生成契约或重新分配执行者
4. 保留当前 `maxReworkAttempts = 3` 的保护阈值，但让超限后的失败语义更明确，尽量转为人工介入而不是裸错误。

**测试与验证：**

- `cd backend && go test ./internal/orchestrator -run Rework -v`
- `cd backend && go test ./internal/service ./internal/orchestrator -v`

**验收标准：**

- 返工路径中可观察到 `supervisor -> worker -> reviewer -> supervisor -> worker -> reviewer`
- 每次返工都带有明确 `rework_brief`
- 超过返工上限时不会静默失败

**建议提交信息：**

```bash
git commit -m "feat: reroute rework through supervisor"
```

## 8. 执行时的技术要求

实施过程中需额外遵守以下要求：

1. 后端每个工作包优先采用“先写失败测试，再做最小实现”的方式推进。
2. `reviewpolicy` 必须单独封装，不要把模板定义散落到 `service`、`runtime`、`frontend` 各层。
3. 首轮若 metadata 存储已经够用，不要提前引入 migration。
4. 返工回主管时，不要顺手改动与本工作包无关的编排语义。
5. 前端展示必须兼容历史 review/approval 数据，不能因为旧数据没有评分卡字段就报错。

### 9.3 手工验收

至少验证以下场景：

2. 一个任务在硬门槛全部通过且总分超过阈值时进入 `approved`。
3. 一个任务在硬门槛失败或总分不足时进入 `needs_revision`。
4. `needs_revision` 后会生成 `rework_brief`，并重新回到主管。
5. 审批页能看到评分摘要，而不是只有“已通过”。
6. 历史旧 review 记录在没有评分卡结构时仍可正常展示。

## 10. 建议执行顺序

建议严格按下面顺序推进：

1. `WP-01` 评分模板与策略注册
2. `WP-02` Reviewer 协议、Prompt 与校验器升级
3. `WP-03` 结构化评审结果落库与审批上下文增强
4. `WP-04` 返工摘要与主管回收链路
5. `WP-05` 前端展示与文档同步

原因如下：

1. 先定义模板，后扩展协议，避免 Reviewer 输出结构反复变化。
2. 先把数据写稳，再修改返工路由，降低调试复杂度。
3. 最后再改前端和文档，避免展示层追着后端结构反复改。

---

## 11. 计划完成标准

当以下条件全部满足时，可认为本实施计划完成：

1. 六类任务都有可用的默认评分模板。
2. Reviewer 输出已升级为结构化评分卡。
3. `review_report.metadata` 中可看到完整评分结果。
4. `needs_revision` 后可生成明确的 `rework_brief`。
5. 返工链路已重新回到责任主管。
6. 前端评审页和审批页都能展示评分摘要。
7. 数据库设计和 API 文档已同步更新。

---

## 12. 结论

本计划采用“先模板、后协议、再落库、再改返工、最后补展示”的顺序推进，目标是在不先扩表的前提下，把统一评审评分卡与返工治理能力完整接入灵筹现有主链路。

这样做可以以最小代价先回答四个关键问题：

1. 为什么打回
2. 打回依据是什么
3. 通过标准是什么
4. 下一轮该怎么改

这也是本计划相对于当前评审闭环最重要的增量价值。
