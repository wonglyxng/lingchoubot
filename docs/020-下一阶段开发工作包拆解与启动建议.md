## 2. 文档目标

在当前基线验收完成并完成第二阶段关键增强后，为下一阶段研发建立一份可执行、可排期、可验收的工作包文档。

## 3. 阶段判断

### 3.1 是否意味着进入下一阶段开发

**是，但需要明确口径。**

从项目治理口径看：

1. [017-MVP验收里程碑文档](017-MVP%E9%AA%8C%E6%94%B6%E9%87%8C%E7%A8%8B%E7%A2%91%E6%96%87%E6%A1%A3.md) 已经把 MVP 定义为“基线冻结版本”。
2. [018-真实LLM接入与Temporal适配层实现总结](018-%E7%9C%9F%E5%AE%9ELLM%E6%8E%A5%E5%85%A5%E4%B8%8ETemporal%E9%80%82%E9%85%8D%E5%B1%82%E5%AE%9E%E7%8E%B0%E6%80%BB%E7%BB%93.md) 与 [019-自动化集成测试补充总结](019-%E8%87%AA%E5%8A%A8%E5%8C%96%E9%9B%86%E6%88%90%E6%B5%8B%E8%AF%95%E8%A1%A5%E5%85%85%E6%80%BB%E7%BB%93.md) 已经完成了“真实执行引擎接入”和“自动化测试补洞”的第一步。
3. 因此当前阶段的主矛盾，已经从“把 MVP 功能做出来”转为“让系统按设计真正执行、可靠执行、可持续回归”。

本文统一使用“下一阶段”口径，与 [docs/003-灵筹 MVP 开发路线图（按周拆解）.md](003-%E7%81%B5%E7%AD%B9%20MVP%20%E5%BC%80%E5%8F%91%E8%B7%AF%E7%BA%BF%E5%9B%BE%EF%BC%88%E6%8C%89%E5%91%A8%E6%8B%86%E8%A7%A3%EF%BC%89.md) 第 16 节和 [docs/017-MVP验收里程碑文档.md](017-MVP%E9%AA%8C%E6%94%B6%E9%87%8C%E7%A8%8B%E7%A2%91%E6%96%87%E6%A1%A3.md) 第 9 节保持一致，不再使用此前那种以 MVP 为中心的临时命名。

从执行口径看：

1. 本文档落地到 `docs/`，表示**下一阶段已进入规划冻结和启动准备**。
2. 当 `WP-N01` 正式排期并开始实现时，可视为**下一阶段开发正式开工**。

结论：**是，项目已经进入下一阶段的规划冻结与启动准备；本文是该阶段的执行基线文档。**

### 3.2 当前阶段的核心主题

下一阶段不再优先扩张页面和表结构，而是集中做五件事：

1. 让组织模型真正驱动编排。
2. 让工作流真正具备持久化执行能力。
3. 让真实 LLM 执行链路具备质量护栏。
4. 让主链路具备 CI 级自动回归能力。
5. 让控制台从轮询演进到实时可观测。

## 4. 总体执行顺序

建议严格按以下顺序推进：

1. `WP-N01` 组织模型对齐与主管链路闭环
2. `WP-N02` Temporal 编排收口与恢复执行
3. `WP-N03` 真实 LLM 质量治理与执行护栏
4. `WP-N04` CI 级端到端回归与发布准入
5. `WP-N05` 前端实时控制台与运行态可观测

其中：

- `WP-N01` 是业务模型收口，必须最先做。
- `WP-N02` 是执行可靠性收口，依赖 `WP-N01` 输出的主链路定义。
- `WP-N03` 和 `WP-N04` 可以在 `WP-N02` 中后段部分并行，但仍应保持先后主次。
- `WP-N05` 放在最后，不是因为不重要，而是它依赖前面四项稳定输出事件和状态。

## 5. 工作包拆解

### WP-N01 组织模型对齐与主管链路闭环

**目标：**

把 [001-灵筹系统总体架构设计.md](001-%E7%81%B5%E7%AD%B9%E7%B3%BB%E7%BB%9F%E6%80%BB%E4%BD%93%E6%9E%B6%E6%9E%84%E8%AE%BE%E8%AE%A1.md) 和 [002-灵筹 Agent 组织树与职责说明.md](002-%E7%81%B5%E7%AD%B9%20Agent%20%E7%BB%84%E7%BB%87%E6%A0%91%E4%B8%8E%E8%81%8C%E8%B4%A3%E8%AF%B4%E6%98%8E.md) 中“PM -> 多主管 -> 执行 Agent -> 评审”的组织模型真正落到编排逻辑中，而不是只停留在数据库字段和查询 API。

**为什么先做：**

当前系统虽然有 `reports_to`、组织树查询和能力字段，但编排主链路仍是通用 `supervisor` / `worker` 路由，尚未形成真正的“主管负责制”。如果不先收口这层模型，后续 Temporal、LLM、实时事件都会建立在过于简化的运行假设上。

**范围：**

1. 定义部门主管角色集和角色语义。
2. 把 `reports_to`、`managed_roles`、`capabilities` 接入编排派单逻辑。
3. 引入升级路径和返工路径。
4. 明确任务从 PM 到主管、从主管到执行者、从评审回主管的责任链。

**非目标：**

1. 不新增复杂多租户权限模型。
2. 不在本工作包中引入 NATS。
3. 不追求一次实现文档 002 中的全部角色，只先实现最小可运行主管集。

**建议最小角色集：**

1. `pm_supervisor`
2. `dev_supervisor`
3. `qa_supervisor`
4. `backend_worker`
5. `frontend_worker`
6. `qa_worker`
7. `reviewer_worker`

**涉及模块：**

- `backend/internal/model`
- `backend/internal/repository/agent.go`
- `backend/internal/service/agent.go`
- `backend/internal/orchestrator/engine.go`
- `backend/internal/orchestrator/temporal_activities.go`
- `backend/internal/runtime`
- `backend/internal/handler/agent.go`
- 前端 Agent / Workflow / Task 相关页面

**任务清单：**

1. 补一份角色矩阵，明确主管和执行者之间的汇报关系、可管理角色、可用工具和升级条件。
2. 在 Agent 模型上补充或固化主管路由所需字段，不再只靠通用 role 字符串驱动。
3. 为编排器增加“按主管域路由”的任务分派逻辑，而不是直接找任意 `supervisor`。
4. 为 Worker 分派增加“主管来源”与“上游责任人”信息。
5. 为评审结果增加“approved / needs_revision -> 后续动作”规则，由主管而非执行者驱动返工。
6. 将 `escalation_policy` 从存储字段接入运行时判断。
7. 补充主管链路、升级路径、返工路径的自动化测试。
8. 更新 Demo 脚本与控制台展示，使组织链路可演示。

**交付物：**

1. 组织角色矩阵文档
2. 新版编排路由逻辑
3. 主管链路自动化测试
4. 更新后的 Demo 数据和演示说明

**验收标准：**

1. 同一个项目中，至少可见 PM、开发主管、测试主管三层责任链。
2. 开发类任务只会进入开发主管链路，不再由通用主管兜底。
3. 评审打回后，任务能回到主管链路重新安排，而不是停在执行者侧。
4. 至少一条升级场景可被记录并回放。

**依赖：** 无

### WP-N02 Temporal 编排收口与恢复执行

**目标：**

把当前已经接入的 Temporal 适配层从“可跑通”提升为“可长期运行”，补齐步骤持久化、恢复执行、审批等待和独立 worker 进程等关键能力。

**为什么第二个做：**

编排可靠性必须建立在稳定的主链路模型上。先完成 `WP-N01` 后，再把该链路迁移或收口到 Temporal，才能避免重复改 workflow 定义。

**范围：**

1. 补齐 Temporal 模式下的 `workflow_step` 写入与查询。
2. 支持等待审批后的恢复执行。
3. 支持失败重试、取消执行、超时终止等最小控制能力。
4. 将 API 进程与 Temporal worker 进程分离。

**非目标：**

1. 不做复杂 Saga 补偿框架。
2. 不做多工作流模板市场。
3. 不做完整事件总线。

**涉及模块：**

- `backend/internal/orchestrator/temporal_engine.go`
- `backend/internal/orchestrator/temporal_workflow.go`
- `backend/internal/orchestrator/temporal_activities.go`
- `backend/internal/orchestrator/temporal_worker.go`
- `backend/internal/service/workflow.go`
- `backend/internal/handler/orchestrator.go`
- `backend/cmd/api`
- 新增或拆分 worker 启动入口

**任务清单：**

1. 定义 Temporal 模式下的标准工作流状态映射。
2. 在每个 Activity 执行前后写入或更新 `workflow_step`。
3. 设计审批等待点，允许工作流暂停并在审批完成后恢复。
4. 增加 run 取消、失败重试、超时标记接口或服务能力。
5. 将 Temporal worker 从 API 进程解耦，形成单独进程或启动模式。
6. 补齐 Temporal 模式查询接口与错误处理。
7. 为 Temporal 主流程增加自动化测试和手工验收脚本。

**交付物：**

1. 可独立启动的 Temporal worker
2. 带步骤持久化的 Temporal 主流程
3. 暂停 / 恢复 / 取消的最小控制能力
4. Temporal 模式测试脚本

**验收标准：**

1. Temporal 模式下可完整查询 run 和 step。
2. API 进程重启不会丢失运行中的流程控制能力。
3. 至少一条需要审批的流程可暂停并恢复。
4. 本地编排模式仍可作为 fallback 保留，不破坏现有 API。

**依赖：** `WP-N01`

### WP-N03 真实 LLM 质量治理与执行护栏

**目标：**

把已经接入的真实 LLM 从“能调用模型”提升为“可控、可测、可回归”的执行能力。

**为什么第三个做：**

如果没有质量护栏，真实 LLM 接入只会让系统不稳定；而在组织模型和编排可靠性稳定后，再做质量治理，测试和评估才有稳定上下文。

**范围：**

1. 提示词版本管理。
2. LLM 输出结构校验与失败降级。
3. 按角色的模型预算、耗时、成本统计。
4. 构建一套固定评测样本，验证 PM / Supervisor / Worker / Reviewer 的输出质量。

**非目标：**

1. 不引入向量知识库。
2. 不构建通用 Prompt 平台。
3. 不做复杂多模型自动博弈。

**涉及模块：**

- `backend/internal/runtime/llm_client.go`
- `backend/internal/runtime/llm_runner.go`
- `backend/internal/runtime/llm_prompts.go`
- `backend/internal/config/config.go`
- `backend/internal/orchestrator`
- `backend/internal/gateway`
- 审计和 tool_call 相关模块

**任务清单：**

1. 为系统提示词建立版本字段和变更记录。
2. 为 LLM 输出增加 schema 校验和兜底修复逻辑。
3. 为每次 LLM 调用补充耗时、模型、token、错误原因等审计信息。
4. 增加每个角色的失败降级策略，例如回退到 Mock Runner 或进入人工审批。
5. 整理一组固定评测输入样本，覆盖项目分解、契约生成、工件生成、评审结论四类场景。
6. 为关键模型升级建立回归评估流程。

**交付物：**

1. 提示词版本治理方案
2. LLM 结构化输出校验层
3. 角色级成本 / 质量评估报告
4. 固定评测样本集

**验收标准：**

1. 任一角色的 LLM 响应不合格时，系统不会静默写入错误结果。
2. 每次 LLM 调用都可追踪模型、耗时、失败原因和输出质量。
3. 模型切换前后，可通过固定样本对比质量变化。
4. 至少一条真实 LLM 主链路在回归样本上稳定通过。

**依赖：** `WP-N01`、`WP-N02`

### WP-N04 CI 级端到端回归与发布准入

**目标：**

把当前以单元测试和手工 Demo 为主的验证方式，升级为面向关键业务链路的 CI 级自动回归体系。

**为什么第四个做：**

当前 019 已经补了 runtime、config、handler 层测试，但还没有“完整项目闭环”的自动验收。前面三项如果没有回归兜底，后续每次改编排或改模型都会增加回归风险。

**范围：**

1. 建立本地编排模式的主链路集成测试。
2. 建立 Temporal 模式的主链路集成测试。
3. 建立 MinIO、审批、返工、工具调用等关键场景回归。
4. 将这些测试接入 CI，作为合并准入条件。

**非目标：**

1. 不要求一次做满前端 E2E 浏览器自动化。
2. 不要求所有 API 都有黑盒测试。
3. 不做性能压测体系。

**涉及模块：**

- `backend/internal/orchestrator`
- `backend/internal/runtime`
- `backend/internal/gateway`
- `backend/internal/handler`
- Demo 脚本和 CI 配置
- 可能新增 `backend/test/` 或 `backend/integration/` 目录

**任务清单：**

1. 定义关键 happy path 和 unhappy path 列表。
2. 补充 Engine 本地模式的黑盒集成测试。
3. 补充 Temporal 模式下的集成测试。
4. 补充 MinIO 实际上传与审批拒绝返工路径测试。
5. 将关键测试拆分为快速集和完整集，分别用于 PR 与主干回归。
6. 将回归结果接入文档或 CI 报告。

**建议最小场景集：**

1. Happy path：项目 -> 阶段 -> 任务 -> 工件 -> 评审 -> 审批通过
2. Return path：评审打回 -> 主管重分派 -> 重新执行 -> 审批通过
3. Approval reject path：审批拒绝 -> 任务回退 -> 再次提交
4. Tool path：工具调用被允许 / 被拒绝 / 被升级
5. Storage path：工件写入 MinIO 并落版本记录

**交付物：**

1. 主链路自动化回归用例
2. CI 准入规则
3. 回归说明文档

**验收标准：**

1. 至少 5 类关键场景具备可重复自动执行的回归用例。
2. PR 合并前至少执行快速回归集。
3. 主干分支至少执行完整回归集。
4. 本地 Engine 与 Temporal Engine 共享同一套主业务验收断言。

**依赖：** `WP-N02`、`WP-N03`

### WP-N05 前端实时控制台与运行态可观测

**目标：**

把当前以轮询为主的控制台，升级为可实时展示工作流、审批、工具调用和审计轨迹的运行态界面。

**为什么第五个做：**

前端实时化的价值建立在前面四个工作包稳定输出状态和事件的基础上。先做实时传输而没有稳定的后端事件语义，只会把前端复杂度前置。

**范围：**

1. 为 workflow、approval、tool_call、audit 提供服务端推送能力。
2. 控制台关键页面从轮询切到 SSE 优先、轮询兜底。
3. 增加运行态调试视图，如当前活动、等待审批、最近失败步骤、最近 LLM 调用摘要。

**非目标：**

1. 不构建复杂运维大盘。
2. 不引入过重的实时消息中间件。
3. 不在本阶段做前端权限系统重构。

**涉及模块：**

- `backend/internal/handler`
- `backend/internal/service`
- 可能新增事件推送层
- `frontend/src/app/workflows`
- `frontend/src/app/approvals`
- `frontend/src/app/tool-calls`
- `frontend/src/app/audit`
- `frontend/src/lib/api.ts`

**任务清单：**

1. 定义控制台实时事件模型。
2. 选择 SSE 作为第一优先实现方式，轮询保留兜底。
3. 为关键页面增加实时订阅和状态同步。
4. 增加运行中、等待审批、失败、恢复等状态的界面表达。
5. 补充前端交互验证和手工演示脚本。

**交付物：**

1. 实时事件接口
2. SSE 驱动的工作流 / 审批 / 工具调用页面
3. 运行态可观测视图

**验收标准：**

1. 工作流状态变化可在前端实时更新，无需刷新页面。
2. 审批完成、工具调用完成、关键审计事件可实时出现。
3. SSE 断开后系统能自动降级回轮询，不影响主流程操作。
4. 控制台可直接演示“运行中 -> 等待审批 -> 恢复 -> 完成”的完整过程。

**依赖：** `WP-N02`、`WP-N04`

## 6. 推荐里程碑切分

为了避免一次性铺得太大，建议把这五个工作包拆成三个可验收里程碑：

### M1 组织与编排基线

包含：

1. `WP-N01`
2. `WP-N02`

退出条件：

1. 新组织链路已接管主编排。
2. Temporal 模式下可查询完整 run / step。
3. 审批等待与恢复至少打通一条主链路。

### M2 真实执行质量基线

包含：

1. `WP-N03`
2. `WP-N04`

退出条件：

1. 真实 LLM 调用具备审计和质量校验。
2. 关键场景进入 CI 级回归。
3. 模型与提示词调整具备回归依据。

### M3 实时控制台基线

包含：

1. `WP-N05`

退出条件：

1. 工作流、审批、工具调用、审计具备实时展示能力。
2. 轮询退为兜底手段。
3. 控制台可承载完整演示和问题定位。

## 7. 建议的启动动作
