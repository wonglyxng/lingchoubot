# 035-工作流阶段审批门禁与自动恢复实现总结

## 1. 变更摘要

本次实现将默认本地编排引擎的审批流从“所有阶段一路跑到待审批”调整为“阶段内可并行待批，阶段间必须等待上一阶段审批收口后再继续”。

新增 `workflow_run.status = waiting_approval`，工作流在当前阶段评审通过并生成审批后会停在审批关口；当同一阶段审批全部收口后，系统会基于审批记录中的 `run_id` 自动恢复执行。若审批被拒绝导致任务进入 `revision_required`，恢复后会回到当前阶段继续返工，而不会直接进入后续阶段。

## 3. 关键实现说明

### 3.1 工作流新增等待审批状态

- 在 `workflow_run` 状态模型中新增 `waiting_approval`。
- 新增 migration `000016_workflow_waiting_approval`，扩展数据库约束。
- `WorkflowService` 新增 `WaitForApproval` 与 `ResumeRun`，用于持久化暂停/恢复状态。

### 3.2 本地引擎按阶段顺序推进

- `Engine` 新增 `continueRun` 与 `ResumeRun`。
- `continueRun` 会按 `project_phase.sort_order` 逐阶段推进。
- 同阶段任务仍允许一起执行并一起进入 `pending_approval`。
- 只要当前阶段存在 `pending_approval` 任务，运行就会停在 `waiting_approval`，不会继续处理下一个阶段。
- 阶段状态会在推进过程中从 `pending -> active -> completed` 更新。

### 3.3 审批收口后自动恢复

- `ReviewReportService` 在自动创建审批请求时，将 `run_id` 与 `phase_id` 写入 `approval_request.metadata`。
- `ApprovalRequestService` 在审批决策后检查当前阶段是否还有待审批任务。
- 若当前阶段审批已收口，则自动触发对应工作流恢复执行。
- 若该阶段还有返工任务，恢复后会继续当前阶段；若该阶段已全部完成，则进入下一阶段。

### 3.4 前端与文档同步

- 工作流页面新增 `waiting_approval` 状态文案与提示说明。
- 工作流详情在等待审批时会继续轮询，避免审批收口后页面状态滞后。
- 数据库设计与 API 文档补充了阶段门禁和自动恢复说明。

## 4. 运行方式

1. 运行数据库 migration：

```bash
cd backend
go run ./cmd/migrate -direction up
```

2. 启动后端 API：

```bash
cd backend
go run ./cmd/api
```

3. 启动前端：

```bash
cd frontend
npm install
npm run dev
```

4. 在控制台启动一个项目工作流，观察工作流运行状态先进入 `等待审批`，审批收口后再继续推进到下一阶段。

## 6. 已知限制

- 本次闭环当前落在默认本地编排引擎路径；Temporal 路径尚未同步实现相同的阶段暂停/恢复能力。
- 当前门禁粒度是“阶段级串行、阶段内并行”，尚未引入任务级依赖图；例如同一阶段内部若还需要严格的前后置关系，后续需要补 `depends_on` 一类模型与校验。
