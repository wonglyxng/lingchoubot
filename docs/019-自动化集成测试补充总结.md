# 019-自动化集成测试补充总结

## 1. 变更摘要

为 LLM Agent 接入层、配置模块、编排 Handler 补充了完整的自动化测试，新增 4 个测试文件共计 **36 个测试用例**，覆盖了此前完全无测试的三个关键模块：runtime/LLM、config、handler/orchestrator。

## 3. 关键实现说明

### 3.1 LLM Client 测试 (llm_client_test.go)
- 使用 `httptest.NewServer` 创建本地 mock LLM 服务器
- 验证请求格式（Authorization header、Content-Type、request body 结构）
- 覆盖场景：成功响应、HTTP 错误码（429）、API 错误、空 choices、非法 JSON、服务不可达
- 验证默认配置（Temperature=0.3, MaxTokens=4096）和自定义配置

### 3.2 LLM Runner 测试 (llm_runner_test.go)
- 测试 `LLMAgentRunner.Execute()` 在各种 LLM 响应下的行为
- 覆盖场景：正常 JSON 输出、LLM 调用失败（返回 failed status）、非法 JSON 响应、空 status 字段自动补全
- 测试 `RegisterLLMRunners` 注册逻辑：验证 4 个角色 + 3 个专业化 worker 都正确注册
- 验证 per-role client 分配：PM 使用独立 client，其他角色回退到默认 client
- 测试 `buildSystemPrompt` 和 `buildUserPrompt` 的输出正确性

### 3.3 Config 测试 (config_test.go)
- 测试 `ResolveForRole` 在五种场景下的行为：全局默认、部分覆盖、完全覆盖、未知角色、nil Roles
- 测试 `Load()` 函数的环境变量读取和默认值
- 测试 `loadLLMConfig` 的 per-role 环境变量解析（LLM_PM_MODEL 等）
- 测试 `DatabaseConfig.DSN()` 输出格式
- 使用 `os.Setenv/Unsetenv` 隔离环境变量影响

### 3.4 Orchestrator Handler 测试 (orchestrator_test.go)
- 实现 `mockWorkflowEngine` mock，满足 `orchestrator.WorkflowEngine` 接口
- 通过 `httptest.NewRecorder` + `http.NewServeMux` 模拟完整 HTTP 请求 → 响应流程
- 覆盖 StartRun（4 个用例）：成功、缺少 project_id、非法 body、引擎错误
- 覆盖 GetRun（2 个用例）：成功、不存在
- 覆盖 ListRuns（3 个用例）：成功、默认 limit、引擎错误

## 5. 运行方式

```bash
# 运行所有测试
cd backend && go test ./...

# 运行新增的 LLM 相关测试
go test ./internal/runtime/ -v -run "TestLLMClient|TestLLMRunner|TestRegisterLLM|TestBuild"

# 运行配置测试
go test ./internal/config/ -v

# 运行编排 Handler 测试
go test ./internal/handler/ -v -run "TestOrchestrator"
```

## 6. 测试方式

所有测试为纯单元/集成测试，不依赖外部服务（数据库、LLM API、Temporal）：
- LLM 测试使用 `httptest.NewServer` 模拟完整 HTTP 交互
- Handler 测试使用 `mockWorkflowEngine` 模拟编排引擎
- Config 测试通过 `os.Setenv` 控制环境变量

```bash
go test ./... -v
```

## 7. 已知限制

1. **Engine.Run() 端到端测试未覆盖**：由于 Engine 依赖大量 service 层（均使用具体 `*repository.XXRepo` 类型而非接口），完整的 Engine.Run() 测试需要用 sqlmock 模拟 30+ 条 SQL 查询，维护成本极高。建议后续将 service 层的 repo 依赖改为接口类型以支持更轻量的 mock。
2. **Temporal 适配层未测试**：temporal_engine.go / temporal_workflow.go / temporal_activities.go 依赖 Temporal SDK client，需要 Temporal 测试环境才能有效测试。
3. **Mock Agent 测试未增加**：6 个 mock agent 文件（mock_pm.go 等）逻辑简单且已被 `registry_test.go` 间接覆盖。
