## 2. 通用约定

### 2.1 统一响应格式

所有 API 返回统一 JSON 结构：

**成功：**
```json
{
  "success": true,
  "data": { ... },
  "error": null
}
```

**失败：**
```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "ERROR_CODE",
    "message": "human readable message"
  }
}
```

### 2.2 常见业务错误码（节选）

| HTTP | error.code | 说明 |
|------|------------|------|
| 400 | `INVALID_BODY` | JSON 解析失败 |
| 400 | `CREATE_FAILED` / `UPDATE_FAILED` / `TRANSITION_FAILED` | 业务校验或写入失败 |
| 409 | `ROLE_CODE_CONFLICT` | Agent 的 `role_code` 与现有记录冲突 |
| 404 | `NOT_FOUND` | 资源不存在 |
| 500 | `QUERY_FAILED` / `DELETE_FAILED` | 查询或删除异常 |
| 503 | `DB_UNAVAILABLE` | 数据库不可达（readyz） |

### 3.1 健康检查（无 `/api/v1` 前缀）

#### GET /healthz

检查 API 进程是否存活。

**响应示例：**
```json
{
  "success": true,
  "data": { "status": "ok" },
  "error": null
```

检查数据库是否可达。

#### GET /api/v1/ping

连通性探测，返回 `message`、`version`。

### 3.4 Agent

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/agents` | 注册 |
| GET | `/api/v1/agents` | 列表（`limit` / `offset`） |
| GET | `/api/v1/agents/{id}` | 详情 |
| PUT | `/api/v1/agents/{id}` | 更新 |
| DELETE | `/api/v1/agents/{id}` | 删除 |

**角色 `role`：** `pm` \| `supervisor` \| `worker` \| `reviewer`。`reports_to` 为上级 Agent 的 UUID，可空。

**关键字段：** `role_code`（组织角色编码，非空时必须唯一）、`agent_type`、`specialization`、`managed_roles`（数组）、`allowed_tools`（数组）、`risk_level`、`metadata`。

**类型 `agent_type`：** `llm` \| `human` \| `mock`。控制台“新建 Agent”默认使用 `llm`，不再暴露 `mock` 选项；`mock` 仅保留给历史数据兼容或测试场景。

**LLM 元数据：** 当 `agent_type = llm` 时，服务端会在 `metadata.llm` 中维护模型配置：

```json
{
  "metadata": {
    "llm": {
      "provider": "openai",
      "model": "gpt-4.1-mini"
    }
  }
}
```

支持的 `provider`：`openai`、`deepseek`、`qwen`、`moonshot`、`zhipu`、`siliconflow`、`openrouter`、`ollama`。未提供 `metadata.llm` 时，服务端默认写入 `openai / gpt-4.1-mini`。

**冲突语义：** 当创建或更新 Agent 时，如果 `role_code` 与已有 Agent 重复，接口返回 `409 ROLE_CODE_CONFLICT`。

**创建请求示例：**

```json
{
  "name": "后端执行者-01",
  "role": "worker",
  "agent_type": "llm",
  "specialization": "backend",
  "description": "负责后端实现与修复",
  "metadata": {
    "llm": {
      "provider": "deepseek",
      "model": "deepseek-chat"
    }
  }
}
```

### 3.5 任务 `task`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/tasks` | 创建 |
| GET | `/api/v1/tasks` | 列表；查询参数：`project_id`、`phase_id`、`status`、`limit`、`offset` |
| GET | `/api/v1/tasks/{id}` | 详情 |
| PUT | `/api/v1/tasks/{id}` | 更新（若变更 `status`，走与 `TransitionStatus` 相同的状态机规则） |
| PATCH | `/api/v1/tasks/{id}/status` | 仅变更状态；body：`{ "status": "<TaskStatus>" }` |
| DELETE | `/api/v1/tasks/{id}` | 删除 |

**任务状态：** `pending`、`assigned`、`in_progress`、`in_review`、`pending_approval`、`revision_required`、`completed`、`failed`、`cancelled`、`blocked`。合法流转由服务端 `TaskStatus.CanTransitionTo` 校验；非法转换返回 `TRANSITION_FAILED`。

> **评审→审批联动：** 当评审报告 verdict 为 `approved` 时，系统自动将任务从 `in_review` 推进到 `pending_approval`，并创建对应的 `approval_request`。审批通过后任务进入 `completed`，审批拒绝后进入 `revision_required`。

> **阶段门禁：** 默认本地编排引擎下，同一阶段内的任务可以并行进入 `pending_approval`；但跨阶段推进会受 `project_phase.sort_order` 约束。工作流在当前阶段审批未收口前会停在 `workflow_run.status = waiting_approval`，不会继续生成后续阶段的待审批项。

**JSON 字段：** `input_context`、`output_summary`、`metadata` 均为对象（JSONB）。

### 3.12 评审报告 `review_report`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/reviews` | 创建评审报告（verdict 为 rejected/needs_revision 时自动打回任务） |
| GET | `/api/v1/reviews` | 列表；查询参数：`run_id`、`task_id`、`reviewer_id`、`verdict`、`limit`、`offset` |
| GET | `/api/v1/reviews/{id}` | 详情 |

**创建请求体：** `task_id`（必填）、`reviewer_id`（必填）、`verdict`（必填：`approved` / `rejected` / `needs_revision`）、`run_id`（可选，工作流自动写入）、`artifact_version_id`（可选）、`summary`、`findings`（数组）、`recommendations`（数组）、`metadata`（可选）。

> **评审联动：** 当 verdict 为 `approved` 时，系统自动将任务状态从 `in_review` 推进到 `pending_approval`，并自动创建关联的审批请求；当 verdict 为 `rejected` 或 `needs_revision` 时，系统自动将任务状态流转为 `revision_required`（要求任务当前处于 `in_review` 状态）。

> **评审上下文：** 工作流自动生成的评审报告会在 `metadata` 中写入 `artifact_count` 与 `artifacts[]`，用于明确说明本次评审基于哪些交付物执行，避免出现“0 个交付物已通过评审”的歧义。

### 3.13 审批 `approval_request`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/approvals` | 发起审批请求 |
| GET | `/api/v1/approvals` | 列表；查询参数：`project_id`、`task_id`、`status`、`limit`、`offset` |
| GET | `/api/v1/approvals/{id}` | 详情 |
| POST | `/api/v1/approvals/{id}/decide` | 审批决策 |

**创建请求体：** `project_id`（必填）、`requested_by`（必填，Agent UUID）、`title`（必填）、`task_id`（可选）、`artifact_id`（可选）、`description`、`approver_type`（默认 `user`）、`approver_id`。

**决策请求体：** `{ "status": "approved" | "rejected", "note": "审批意见" }`

> **审批回写：** approved 时自动将关联任务推进为 `completed`（要求任务处于 `pending_approval`）；rejected 时推为 `revision_required`。

> **自动恢复：** 默认本地编排引擎下，当同一阶段内不再存在 `pending` 状态的审批请求时，工作流会根据 `approval_request.metadata.run_id` 自动恢复执行：若本阶段仍有 `revision_required` 任务，则回到当前阶段继续返工；若本阶段任务均已完成，则进入下一个阶段。

> **审批展示上下文：** 对于由评审自动生成的审批请求，`metadata` 会补充 `review_id`、`review_summary`、`findings`、`recommendations`、`artifact_count`、`artifacts[]`，审批页据此展示完整审批依据。

## 4. 工作流编排（第 6 周）

### 4.1 启动工作流运行

`POST /api/v1/orchestrator/runs`

启动一个项目的完整工作流：PM 分解 → 主管规划 → 执行生产 → 评审验收。

> **启动前校验：** 若缺少活跃的 `pm`、`supervisor`、`worker`、`reviewer` 任一角色，接口返回 `PRECHECK_FAILED`，不会创建新的 `workflow_run`。

> **阶段推进规则：** 默认本地编排引擎按 `project_phase.sort_order` 顺序推进。一个阶段内的任务可并行处理并并行待批；只有当该阶段审批全部收口后，运行才会从 `waiting_approval` 恢复到下一阶段。

**请求体：**
```json
{
  "project_id": "uuid"
}
```

**响应：** 返回 `WorkflowRun` 对象，包含所有步骤及其状态。

**运行流程：**
1. PM Agent 读取项目信息，分解为阶段和任务
2. 对每个任务：Supervisor Agent 创建契约并分派 Worker
3. Worker Agent 执行任务，产生工件和交接快照
4. Reviewer Agent 评审交付物，产生评审报告
5. 评审通过后任务进入 `pending_approval`，自动生成审批请求
6. 所有操作产生审计日志

### 4.2 查询运行列表

`GET /api/v1/orchestrator/runs`

### 4.3 查询运行详情

`GET /api/v1/orchestrator/runs/{id}`

**响应：** 返回包含所有步骤详情的 `WorkflowRun` 对象。

## 8. 为何曾出现「任务完成但文档未落地」

根因：**`AGENTS.md` §15 要求 API / 表结构变更后必须同步 `docs/`，§16 将「文档已同步」列入完成定义**，但此前编码迭代在合并功能后**未执行文档更新**，仓库内也不存在自动生成 OpenAPI / 库表文档的流水线，因此 `005` / `004` 仍停留在「仅健康检查 / 待实现表」的描述。

后续约定：合并影响 API 或 migration 的 PR 时，**同时更新** `docs/005-API 接口文档.md` 与 `docs/004-数据库设计.md`（或拆分的专门文档），并在变更记录表中记一行。
