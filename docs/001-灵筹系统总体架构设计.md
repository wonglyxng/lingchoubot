## 2. 背景与目标

灵筹旨在实现一种"组织化多智能体编排系统"：

- 用户只需下达项目级目标；
- 项目负责人 Agent 负责理解需求、制定计划、召集各部门主管 Agent；
- 各部门主管 Agent 继续拆解任务并分派给执行 Agent；
- 系统最终输出可验证的交付物，而不是仅输出自然语言结论；
- 用户以"审批/验收"的方式参与关键节点，而不是介入每一个细节步骤。

---

## 3. 设计目标

灵筹必须满足以下目标：

1. **项目驱动** — 顶层对象是"项目"，不是"聊天会话"。
2. **组织驱动** — 支持"项目负责人 -> 部门主管 -> 执行 Agent"的层级式协同。
3. **工件驱动** — 每一步推进都必须产生明确工件（PRD、架构设计、代码、测试报告等）。
4. **状态机驱动** — 所有任务都必须具备可追踪状态，不依赖自由对话推进。
5. **审批可控** — 高风险动作必须可配置为人工审批。
6. **审计可回放** — 能追踪"谁在什么上下文下，调用了什么工具，生成了什么结果"。
7. **可恢复** — 支持长流程中断恢复、失败重试、等待人工审批后的继续执行。

---

## 4. 非目标

当前阶段不将以下内容作为目标：

1. 不追求完全通用的"万能 Agent 操作系统"；
2. 不优先支持人与 Agent 的高频对话协作体验；
3. 不以"高度拟人化、自由讨论式 AI 公司模拟"为目标；
4. 不在 V0 阶段支持多租户、强隔离 SaaS 能力；
5. 不在 V0 阶段解决所有安全问题，采用"最小权限 + 明确审批 + 全量审计"的工程路线。

---

## 5. 核心设计原则

### 5.1 项目优先于会话

`Project -> Phase -> Task -> Artifact`，而不是 `Chat -> Message -> Reply`。

### 5.2 工件优先于对话

对话只是中间手段，真正决定系统推进的是工件是否满足验收标准。

### 5.3 主管优先于执行 Agent 直连

执行 Agent 的结果应先提交给对应主管 Agent，由主管做初步验收与汇总。

### 5.4 最小权限原则

每个 Agent 只拿到完成当前任务所需的最小工具与最小资源权限。

### 5.5 长流程可恢复

系统面向项目交付，很多流程天然是长生命周期流程，必须支持中断恢复。

### 5.6 任务契约优先于模糊执行

任务在执行前应先明确"做什么、做到什么程度算完成、如何验证"。任务契约至少包括：

- 本轮范围与非目标
- 验收标准与验证步骤
- 可使用工具与升级条件

### 5.7 独立评审优先于执行 Agent 自评

执行 Agent 可以做自检，但关键任务应默认采用"执行者"与"评审者"分离的模式。

### 5.8 结构化交接优先于长对话续跑

关键执行轮次须产出结构化交接信息（已完成项、未完成项、风险、下一步建议、关键工件引用），确保 Agent 切换或会话重置后仍能继续推进。

## 6. 系统定位

> **面向复杂项目交付的多智能体组织操作系统**

核心能力：组织建模、Agent 注册与调度、任务拆解与派发、任务契约与交接管理、工件管理、独立评审与质量门、权限与审批、审计与回放、工具网关。

## 7. 总体架构概览

灵筹整体分为五层（MVP 实际落地层）+ 两层（后续演进层）：

```text
┌──────────────────────────────────────────────────┐
│                 用户与控制台层                     │
│   Web Console / CLI / API / 审批中心 / 看板        │
└──────────────────────────────────────────────────┘
                       │
┌──────────────────────────────────────────────────┐
│                  控制平面层                        │
│  Project Service / Agent Registry / Policy        │
│  Task Orchestrator / Approval Engine              │
└──────────────────────────────────────────────────┘
                       │
┌──────────────────────────────────────────────────┐
│               Agent 运行时层                      │
│  PM Agent / Supervisor Agents / Worker Agents     │
│  统一协议 + Go Runtime（MVP 阶段）                 │
└──────────────────────────────────────────────────┘
                       │
┌──────────────────────────────────────────────────┐
│                  工具执行层                        │
│  Tool Gateway / Git / Shell / Test Runner         │
│  Docs / Storage / External APIs                   │
└──────────────────────────────────────────────────┘
                       │
┌──────────────────────────────────────────────────┐
│                数据与存储层                        │
│  PostgreSQL / S3(MinIO)                           │
└──────────────────────────────────────────────────┘

--- 以下为后续演进层，MVP 阶段不实现 ---

┌──────────────────────────────────────────────────┐
│              工作流编排层（阶段 2）                 │
│  Temporal Workflows / Activities / Sagas          │
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│              事件总线层（阶段 2）                   │
│  NATS Core / JetStream / Domain Events            │
└──────────────────────────────────────────────────┘
```

**MVP 阶段说明：**
- 工作流编排在 MVP 阶段以应用内最小编排器实现，保留 adapter 抽象以便后续切换 Temporal
- 事件总线在 MVP 阶段不引入，领域事件通过数据库 outbox 模式或直接同步调用处理
- Agent Runtime 在 MVP 阶段只需 Go 单语言实现

## 8. 逻辑架构分层说明

### 8.1 用户与控制台层

提供人类操作入口：创建项目、下达目标、配置约束、查看状态、审批高风险动作、预览工件、回放执行过程。

实现形态：
- **Web Console**：Next.js + React + TypeScript
- **API**：REST（MVP 阶段）

### 8.2 控制平面层

灵筹的核心控制面，职责包括：

- 项目生命周期管理
- Agent 注册与发现
- 组织树管理
- 任务分派与升级
- 任务契约管理
- 交接快照索引
- 评审调度与质量门控制
- 权限决策与审批路由
- 工件索引
- 对外暴露统一 API

设计要求：
1. 控制平面是长期运行的中心服务
2. 所有 Agent 都通过控制平面完成注册、鉴权、任务收发
3. 控制平面不直接承担复杂工具执行，负责编排和治理
4. 所有高价值元数据必须先进入控制平面再入库

### 8.3 Agent 运行时层

具体"干活"的 Agent 所在层。分三类：

**管理型 Agent（Supervisor）**
- 项目负责人 Agent、部门主管 Agent
- 负责任务拆解、验收、汇总、升级
- 工具权限相对受限

**执行型 Agent（Worker）**
- PRD Agent、开发 Agent、API 设计 Agent 等
- 职责单一、上下文更聚焦
- 工具权限更具体

**评审型 Agent（Worker，评审模式）**
- 测试 Agent、评审 Agent
- 与执行 Agent 职责分离
- 以任务契约和验收标准为判断依据
- 优先通过工具验证

**MVP 阶段：** 统一使用 Go 实现 Agent Runtime，后续可扩展 Python / TypeScript Runtime。

### 8.4 工具执行层

所有工具通过 Tool Gateway 暴露，Agent 不直接调用外部资源。

**MVP 初期支持：**
- 文档生成器
- 工件存储（MinIO）
- 测试运行器
- 受限 Shell（白名单命令）
- Git 仓库（只读或 feature branch 写）

**后续扩展：**
- 浏览器自动化、CI/CD 触发器、Figma、Jira/Linear、云平台 API

设计要求：
1. 工具必须声明权限范围
2. 每次调用必须被审计
3. 高风险工具必须纳入审批策略
4. 工具调用结果必须结构化回传

### 8.5 数据与存储层

| 组件 | 用途 | MVP 阶段 |
|------|------|----------|
| PostgreSQL | 主业务数据库 + JSONB 半结构化数据 | 必须 |
| S3/MinIO | 工件与大文件存储 | 必须 |
| Redis | 缓存、短期锁 | 可选，MVP 暂不引入 |
| 向量库 | 长期知识检索 | MVP 后 |
| Loki/Prometheus | 日志与指标 | MVP 后 |

**数据分层：**

- **关系型数据（PostgreSQL）**：项目、阶段、任务、Agent、审批单、权限策略、工具注册表
- **半结构化数据（JSONB）**：上下文包、Prompt 快照、模型调用元数据、工具调用请求/响应、任务执行摘要
- **对象存储（MinIO）**：PRD 文件、架构图、代码快照、测试报告、构建产物

## 9. 组织模型设计

灵筹采用四层组织模型：

```text
第 0 层：用户（最终审批人）
第 1 层：项目负责人 Agent
第 2 层：部门主管 Agent（产品 / 架构 / 开发 / 测试 / 发布）
第 3 层：执行 Agent（PRD / API 设计 / 后端开发 / 前端开发 / 测试 等）
```

设计原则：
1. 用户默认只与项目负责人 Agent 交互
2. 部门主管 Agent 负责本部门任务拆解与验收
3. 执行 Agent 不直接决定项目方向
4. 任意 Agent 都必须归属一个明确角色
5. 任意任务都必须有唯一责任主体

> 详细的 Agent 角色定义见 `002-灵筹 Agent 组织树与职责说明.md`

## 10. 核心业务对象模型

### 10.1 Project（项目）

项目是系统顶层对象，包含：项目目标、约束、阶段、负责人 Agent、预算、风险等级、当前状态。

### 10.2 Phase（阶段）

项目内的宏观推进单元：需求澄清、方案设计、开发实现、测试验证、发布交付。

### 10.3 Task（任务）

实际调度单元，支持父子层级。包含：

| 字段 | 说明 |
|------|------|
| 输入上下文 | 任务执行所需的背景信息 |
| 负责人 Agent | 当前任务的执行者 |
| 任务契约 | 范围、非目标、完成定义、验证步骤 |
| 验收标准 | 如何判断任务完成 |
| 权限边界 | 允许使用的工具和资源 |
| 当前状态 | 状态机驱动的流转状态 |
| 重试/超时策略 | 失败处理策略 |

### 10.4 Artifact（工件）

工件是推进依据与审计对象。类型包括：

`prd` / `design` / `api_spec` / `schema_sql` / `source_code` / `task_contract` / `review_report` / `handoff_snapshot` / `test_report` / `deployment_plan` / `release_note`

### 10.5 Approval（审批）

审批单用于承载：高风险动作、阶段门控、预算超限、生产发布、主干合并、权限升级。

## 11. 关键执行流程

### 11.1 项目启动流程

```text
用户创建项目
 -> 控制平面创建 Project
 -> 指定项目负责人 Agent
 -> 项目负责人生成项目章程
 -> 召集各部门主管进行方案会商
 -> 输出阶段计划与任务树
 -> 用户审批立项结果
 -> 进入阶段执行
```

### 11.2 任务执行流程

```text
主管 Agent 创建任务
 -> 明确任务契约与验收方式
 -> Task Orchestrator 记录任务
 -> 任务分派给执行 Agent
 -> 执行 Agent 调用工具产生工件
 -> 执行 Agent 生成交接快照
 -> 工件提交给评审 Agent / 主管 Agent 审核
 -> 评审方按契约执行验证与工具检查
 -> 审核通过则标记完成
 -> 审核失败则打回 revision_required
 -> 如有阻塞则升级为 blocked
```

### 11.3 发布流程

```text
开发任务完成
 -> 测试主管发起回归
 -> 测试通过后生成发布候选包
 -> 发布主管生成部署计划
 -> 触发人工审批
 -> 审批通过后执行发布
 -> 记录发布日志与回滚点
 -> 项目负责人向用户提交交付报告
```

## 12. 技术选型

### 12.1 MVP 阶段技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 控制平面 | Go | 并发模型适合调度，单二进制部署友好 |
| 主数据库 | PostgreSQL | 关系型 + JSONB 混合存储，事务支持完善 |
| 工件存储 | MinIO（S3 兼容） | 本地开发友好，生产可切换 S3 |
| 前端 | Next.js + React + TS | 现代 Web 开发标准栈 |
| 容器编排 | Docker Compose | 本地开发环境 |
| Migration | golang-migrate + 纯 SQL | 简单可控 |

### 12.2 后续演进技术栈

| 组件 | 选型 | 引入时机 |
|------|------|----------|
| 工作流引擎 | Temporal | 阶段 2，MVP 后 |
| 事件系统 | NATS + JetStream | 阶段 2，MVP 后 |
| 缓存 | Redis | 按需引入 |
| 向量库 | pgvector 或独立向量库 | 阶段 3 |
| 可观测性 | Prometheus + Grafana + Loki | 阶段 2 |

## 13. 安全与权限架构

### 13.1 三段权限边界

| 类别 | 说明 | 示例 |
|------|------|------|
| A 类：自动执行 | 无需人工审批 | 读取文档、生成草稿、运行测试、feature branch 代码 |
| B 类：主管审批 | 需部门主管审批 | 修改共享 API、修改数据库 Schema、合并部门工件 |
| C 类：最终审批 | 需用户审批 | 合并主干、正式发布、删除关键数据、外部通知 |

### 13.2 Agent 最小权限模型

每个 Agent 需要声明：可读资源、可写资源、可调用工具、可访问项目范围、是否允许联网、是否允许执行 Shell、是否允许写仓库、是否允许触发 CI/CD。

## 14. 失败恢复设计

| 失败场景 | 处理方式 |
|----------|----------|
| 执行 Agent 崩溃 | 任务进入 `retryable_failed`，重试或迁移到同类 Agent，保留已产生工件 |
| 工具调用失败 | 记录失败原因，按策略自动重试，超限后升级给主管 |
| 人工审批超时 | 进入 `awaiting_human_timeout`，通知项目负责人，支持撤销/延期/转派 |
| 上下文膨胀 | 从最近一次交接快照恢复执行，由控制平面重新装配任务契约 |

## 17. 风险清单

| 风险 | 表现 | 对策 |
|------|------|------|
| 过度自治 | Agent 不断拆任务，系统失控 | 限制递归深度、预算上限、阶段门控审批 |
| 上下文污染 | 不同项目串上下文 | 项目级上下文命名空间、工件显式引用 |
| 工具滥用 | 错误修改仓库、删除数据 | 最小权限、审批闸门、审计记录 |
| 空转讨论 | Agent 大量讨论无产出 | 以工件和状态机作为推进依据 |
| 成本失控 | 大模型循环调用、重复返工 | 任务预算、模型分层、摘要压缩 |
