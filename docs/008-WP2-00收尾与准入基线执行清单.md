## 2. 目的

本清单用于把 `WP2-00 收尾与准入基线` 从阶段级规划拆成可直接执行的工作项。

本工作包的本质不是“继续补功能”，而是给阶段 2 的 Temporal、NATS、多 Agent 和细粒度权限改造建立稳定、可回归、可验证的起跑线。

---

## 3. 当前事实基线

结合当前仓库状态，WP2-00 需要处理的真实缺口如下：

1. 审批拒绝后的任务打回逻辑在 service 层已实现，但 Demo 未覆盖，仍属于“逻辑存在、链路未验证”。
2. `artifact_storage` 工具目前仅返回 `mock://minio/...` URI，尚未接入真实对象存储上传。
3. 后端当前没有自动化测试文件，关键链路仍主要依赖手工或 Demo 验证。
4. 中间件目录仅包含 `cors`、`logging`、`recovery`、`response`，尚无鉴权中间件。
5. 阶段 2 规划已经明确，但缺少一份面向执行的 WP2-00 子任务清单。

结论：WP2-00 必须同时解决“链路补验、存储落地、测试起步、鉴权定稿、文档收口”五件事。

---

## 4. 范围

### 4.1 必做范围

- 审批 `approved` / `rejected` 两条链路都能被端到端验证
- 工件存储从 mock URI 升级为真实 MinIO 上传
- 为审批、工具权限、编排主链路补最小自动化测试
- 输出 API 鉴权设计并确定 MVP 后续接入方式
- 补齐 Demo、使用说明、技术债清单中与 WP2-00 直接相关的内容

### 4.2 不在本工作包内

- Temporal 接入
- NATS / Outbox 接入
- 多 Agent 能力模型升级
- 细粒度授权的完整实现
- 大范围前端重构

这些事项进入后续工作包处理，WP2-00 只负责“准入基线”。

---

## 5. 子任务拆解

### WP2-00-01 审批拒绝链路补测与 Demo 覆盖

**目标：** 验证现有 `approval rejected -> task revision_required` 逻辑在 API 和 Demo 上都成立。

**建议修改模块：**
- `backend/internal/service/approval_request.go`
- `backend/internal/service/task.go`
- `backend/cmd/demo/main.go`
- 新增相关测试文件

**执行内容：**
1. 确认审批拒绝时任务状态流转只通过 service 层完成，不在 handler 层分散处理。
2. 为审批决策补单元测试，覆盖 `approved`、`rejected`、重复决策、非法状态四种场景。
3. 扩展 Demo，至少对一个任务走批准路径、另一个任务走拒绝路径。
4. 在 Demo 输出中明确显示被拒绝任务的最终状态为 `revision_required`。

**交付物：**
- 审批决策测试
- 覆盖拒绝路径的 Demo 输出
- 更新后的验收说明

**验收标准：**
- 审批拒绝后任务状态稳定为 `revision_required`
- Demo 日志可清晰看到 rejected 路径结果
- 审批重复决策被拒绝，且错误行为可测试复现

---

### WP2-00-02 MinIO 实际上传落地

**目标：** 将 `artifact_storage` 从 mock 实现升级为真实对象存储写入。

**建议修改模块：**
- `backend/internal/gateway/artifact_storage.go`
- `backend/internal/config/config.go`
- `backend/cmd/api/main.go`
- `docker-compose.yml`
- 相关文档

**执行内容：**
1. 引入 MinIO 连接配置，包括 endpoint、bucket、access key、secret key、use ssl 等参数。
2. 设计统一的对象存储写入接口，避免工具层直接持有过多 MinIO 细节。
3. 将工具接收到的内容写入 MinIO，并返回真实对象 URI、checksum、size、content type。
4. 若 bucket 不存在，明确是自动创建还是启动前置创建，文档中必须写清。
5. 保留开发模式下的最小降级策略，但不能继续默认 `mock://` 作为成功路径。

**交付物：**
- MinIO 配置项
- 对象存储真实上传实现
- 本地运行说明

**验收标准：**
- 在本地 compose 环境下可成功上传工件
- Tool Gateway 返回真实对象地址或规范化存储标识
- 工件元数据与实际对象内容一致

---

### WP2-00-03 自动化测试基线搭建

**目标：** 让关键链路不再只依赖人工运行 Demo 验证。

**建议修改模块：**
- `backend/internal/service/`
- `backend/internal/gateway/`
- 新增 `_test.go` 文件
- 视情况补测试辅助代码

**执行内容：**
1. 先补单元测试，再决定是否补最小集成测试，不要一开始做庞大测试框架。
2. 单元测试至少覆盖：审批状态变更、任务状态流转、工具权限判断。
3. 集成测试至少覆盖：审批决策 API 或 Demo 主链路中的一个关键片段。
4. 固定测试运行命令，并写入 README 或专门文档。

**最低测试清单：**
- `ApprovalRequestService.Decide`
- `TaskService.TransitionStatus`
- `Gateway.checkPermission`

**交付物：**
- 第一批后端自动化测试文件
- 固定测试命令
- 测试覆盖说明

**验收标准：**
- `go test ./...` 至少能稳定运行核心测试
- 关键状态机行为被自动化验证
- 新增测试不依赖手工预置数据库数据

---

### WP2-00-04 鉴权方案定稿

**目标：** 在编码前先定清楚 API 鉴权模型，避免 WP2-05 时边做边改协议。

**建议修改模块：**
- 文档优先，必要时补 `backend/internal/middleware/` 骨架
- 若只输出方案文档，可暂不接入业务逻辑

**执行内容：**
1. 明确当前系统涉及的身份主体：用户、Agent、内部系统组件。
2. 确定 MVP 后续采用的最小鉴权方式，例如静态 token、JWT、内部 service token，三者选其一并说明原因。
3. 明确哪些 API 在阶段 2 需要受保护，哪些仍可保留公开健康检查。
4. 规划鉴权与工具授权的衔接关系，避免后续重复建模。
5. 给出迁移策略，确保现有 Demo 和前端联调可渐进切换。

**交付物：**
- 鉴权设计文档
- 受保护接口清单
- 迁移策略说明

**验收标准：**
- 能回答“谁可以调用什么 API，以什么凭证调用”
- 方案不与现有 Agent / Tool Gateway 模型冲突
- 后续 WP2-05 可直接据此编码

### WP2-00-05 文档与演示材料收口

**目标：** 把阶段切换前必须留下的文档收齐，避免后续执行失焦。

**建议修改模块：**
- `docs/006-开发进度总结.md`

**执行内容：**
1. 把 WP2-00 的完成状态回写到开发进度总结。
2. 把 Demo 的 approved / rejected 双路径写入演示说明。
3. 将 MinIO 运行说明补入启动文档。
4. 输出一份技术债清单，明确哪些问题被延期到 WP2-01 之后。

**交付物：**
- 更新后的进度文档
- Demo 说明文档
- 技术债清单

**验收标准：**
- 阶段 2 的起始状态在文档中清晰可读
- 新成员可根据文档在本地复现实验环境
- 仍未解决的问题被显式记录，而不是隐含遗留

## 6. 推荐执行顺序

建议按下面顺序推进，减少返工：

1. `WP2-00-04` 鉴权方案定稿
2. `WP2-00-03` 自动化测试基线搭建
3. `WP2-00-01` 审批拒绝链路补测与 Demo 覆盖
4. `WP2-00-02` MinIO 实际上传落地
5. `WP2-00-05` 文档与演示材料收口

说明：
- 鉴权方案先定，避免测试与 Demo 再做一次认证迁移。
- 测试基线先有，后续改 Demo 和上传能力时能即时回归。
- MinIO 上传放在 Demo 双路径之后，避免同时调两条大链路导致定位困难。

---

## 7. 建议交付节奏

如果仍按“一次只做一个明确工作包”的原则推进，建议再拆成以下提交节奏：

1. 第一提交：鉴权设计文档 + 测试基线
2. 第二提交：审批 rejected 链路测试 + Demo 补齐
3. 第三提交：MinIO 接入 + 配置 + 文档
4. 第四提交：进度、技术债、演示材料收口

这样做的好处是：每一步都有独立验收点，且不会把基础设施、业务链路、文档收口混成一个超大提交。

---

## 8. 完成定义

只有满足以下条件，WP2-00 才算完成：

1. Demo 同时覆盖审批通过和审批拒绝两条路径
2. `artifact_storage` 不再以 mock URI 作为默认成功实现
3. 后端已有第一批自动化测试文件并可稳定执行
4. API 鉴权方案已成文并获得确认
5. 相关运行文档、演示说明、技术债清单均已更新

---

## 9. 风险与应对

| 风险 | 表现 | 应对 |
|------|------|------|
| 一次改太多 | Demo、MinIO、鉴权、测试同时开工，回归困难 | 强制按子任务顺序提交 |
| 鉴权方案反复 | 先写代码再定方案，导致中间件返工 | 文档先行，方案先定稿 |
| MinIO 接入扩散 | 把对象存储细节散落到多个模块 | 通过统一存储接口收敛 |
| 测试起步过重 | 一开始搭太复杂的测试框架 | 先围绕 service 和 gateway 做最小闭环 |
| 文档再次滞后 | 代码完成但进度文档不同步 | 将文档收口作为独立子任务 |

---
