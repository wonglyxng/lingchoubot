## 7. 端到端 Demo 演示流程说明

### 7.1 启动方式

```bash
docker compose up --build

cd backend
go run ./cmd/demo -v
```

默认访问地址：

- API：`http://localhost:8080`
- 控制台：`http://localhost:3000`

### 7.2 演示步骤

1. **健康检查**：验证 API 服务与数据库连通性。
2. **注册 Agent 组织树**：创建 PM、Supervisor、Backend Worker、Frontend Worker、Reviewer 共 5 个演示 Agent。
3. **创建 Demo 项目**：通过 API 建立“灵筹 MVP 演示项目”。
4. **启动工作流编排**：运行 PM -> Supervisor -> Worker -> Reviewer 的主链路，自动生成阶段、任务、契约、工件、评审数据。
5. **验证工作流持久化**：查询 `workflow_run` 和 `workflow_step`，确认运行记录已持久化可查。
6. **演示交接快照**：为某个任务创建并查询 `handoff_snapshot`。
7. **验证数据完整性**：检查阶段、任务、工件、评审、组织树等实体数量。
8. **演示审批通过链路**：创建审批请求并执行批准，验证任务自动流转为 `completed`。
9. **演示审批拒绝链路**：创建第二条审批请求并拒绝，验证任务自动回退为 `revision_required`，随后恢复为 `in_progress`。
10. **演示 Tool Gateway**：调用内置工具，验证权限校验、调用记录和工具执行结果。
11. **查询审计时间线**：查看项目级和任务级事件回放。
12. **控制台验收**：在前端页面查看项目详情、任务看板、Agent 组织树、工作流运行、工件、审批和审计记录。

### 7.3 建议演示页面顺序

1. `/`：概览页，确认系统在线状态。
2. `/projects` 与 `/projects/{id}`：查看项目、阶段、任务摘要。
3. `/tasks`：验证状态流转与审批结果。
4. `/agents`：确认组织树与能力信息。
5. `/workflows`：查看工作流运行与步骤。
6. `/artifacts`、`/reviews`、`/handoffs`：查看执行产物、评审结果和交接信息。
7. `/approvals`：展示批准与拒绝两条分支。
8. `/tool-calls`、`/audit`：展示工具调用与全链路审计轨迹。

## 8. 已知限制与技术债清单

| 类别 | 当前限制 / 技术债 | 影响 |
|------|-------------------|------|
| 执行引擎 | 仍以 Mock Agent 为主，尚未接入真实 LLM Agent | 系统已能证明控制平面闭环，但尚未进入真实智能执行阶段。 |
| 编排可靠性 | 当前异步执行仍基于 API 进程内 goroutine；运行记录可持久化，但运行中任务在进程异常退出时无法自动续跑 | 无法满足长期稳定运行与生产级恢复要求。 |
| 工作流引擎 | Temporal 适配层未实现 | 缺少更强的重试、超时、补偿和持久化执行能力。 |
| 事件架构 | Outbox / NATS 未接入 | 关键状态变更仍以内聚式同步调用为主，事件化能力不足。 |
| 实时推送 | 前端仍以轮询为主，尚未切换到 SSE / WebSocket | 实时性与服务端推送效率有限。 |
| 鉴权模型 | 当前为静态 API Key + 工具动作级授权，尚未进入多租户、RBAC、SSO | 可满足 MVP 演示，但不满足复杂组织场景。 |
| 自动化测试 | 现有测试以单元测试为主，缺少完整的端到端自动化集成测试 | 主链路仍部分依赖 Demo 脚本与手工观察验证。 |
| 对象存储 | 已支持 MinIO 实际上传，但对大文件、异常网络、权限边界的集成测试仍不足 | 存储能力可用，但生产稳健性证据不足。 |
| 指标文档 | 部分历史进度文档中的 API / 页面数量为阶段性近似值 | 验收与对外同步时需要统一以当前代码口径为准。 |

## 9. 下一阶段建议路线图

建议后续按“先让系统真正执行，再让系统可靠执行，最后让系统实时可观测”的顺序推进。

| 阶段 | 建议主题 | 核心目标 | 退出条件 |
|------|----------|----------|----------|
| R1 | 接入真实 LLM Agent | 用真实模型替换或补充 Mock Runner，完成任务输入、提示词、工具调用和产出回写闭环 | 至少一条 PM -> Supervisor -> Worker -> Reviewer 主链路由真实 LLM 参与执行并稳定产出工件。 |
| R2 | Temporal 适配层 | 将当前应用内编排迁移到可持久化执行引擎，解决进程重启丢失运行中任务问题 | 标准主流程可由 Temporal 驱动执行，现有 HTTP API 不破坏。 |
| R3 | Outbox + NATS 事件化 | 将关键状态变更异步化、可重试、可审计 | task / artifact / approval 等关键事件可稳定投递并回放。 |
| R4 | 自动化集成测试 | 建立 API 集成测试和端到端回归测试，降低依赖手工 Demo 的风险 | 关键主链路具备可重复执行的 CI 级自动化验证。 |
| R5 | SSE / WebSocket 实时控制台 | 替代轮询，实现工作流、审批、工具调用、审计时间线的实时推送 | 控制台关键页面具备服务端推送能力，轮询退为兜底。 |

建议优先级：**R1 -> R2 -> R4 -> R5**，其中 R3 可在 R2 之后并行推进。

## 10. 里程碑结语

灵筹项目已经完成 MVP 阶段的核心目标：不是“展示一堆孤立模块”，而是交付了一个可运行、可演示、可审计、可扩展的多 Agent 控制平面基线。

从本里程碑开始，项目的主矛盾将从“把功能做出来”转向“把执行质量、恢复能力和真实智能接入做扎实”。因此，本版本应被视为 **MVP 基线冻结版本**，后续迭代应围绕真实执行、可靠编排和实时可观测三个主题展开。
