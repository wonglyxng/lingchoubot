## 关键实现说明

### 测试策略

由于 service 层依赖具体 `*repository.XRepo` 类型（非接口），而 repo 依赖 `*sql.DB`，纯单元测试受限。采用以下策略：

- **Model 层**：直接测试状态常量、状态流转矩阵（含自流转/未知状态）、终态验证
- **Middleware 层**：使用 `httptest.NewRecorder` 测试 JSON/ErrorJSON 函数的响应格式
- **Handler 层**：
  - 使用 nil service 测试 JSON 解析失败路径（不会调到 service）
  - 使用 nil repo 的真实 service 测试业务验证路径（验证在 repo 调用前完成）
  - parsePagination 作为纯函数直接测试

### 测试覆盖的关键路径

- 无效 JSON body → 400 INVALID_BODY
- 缺失必填字段 → 400 CREATE_FAILED
- 分页参数边界值（0/负数/超限/非数字/上界）
- 任务状态流转矩阵（18条有效 + 12条无效 + 9条自流转）
- API 响应格式（success/error envelope、Content-Type、各状态码）


## 已知限制

1. **Service 层无独立单元测试** — service 依赖具体 repo 类型，纯 mock 需引入 sqlmock 或重构为接口。当前通过 handler 层间接覆盖验证逻辑。
2. **Repository 层无单元测试** — 需要真实 PostgreSQL 连接。建议后续引入 testcontainers 或 sqlmock。
3. **Handler happy path 未测试** — 成功创建/查询路径需要完整 service + repo + DB 链路，当前仅测试了错误路径。
4. **前端无自动化测试** — 前端以 TypeScript 编译检查为主，无 Jest/Vitest 测试。
