## 2. 文档目标

为 `WP2-07 组织模型对齐与主管链路闭环` 提供可直接开工的输入基线，明确三件事：

1. 当前实现与目标设计之间的真实差距。
2. 第二阶段应采用的最小组织角色矩阵。
3. 主管链路、路由规则、返工闭环和字段调整方案。

本文不是最终实现总结，而是 `WP2-07` 的设计输入文档。

## 3. 当前实现与主要差距

### 3.1 当前已具备的基础

基于当前代码，系统已经具备以下能力：

1. `agent` 表已支持 `reports_to`、`agent_type`、`specialization`、`capabilities`。
2. Agent 组织树查询和直属下级查询已经可用。
3. Worker 已支持基于任务关键词的专业化路由（backend / frontend / qa）。
4. 编排主链路已稳定支持 `PM -> Supervisor -> Worker -> Reviewer`。
5. `task_contract.escalation_policy` 已有存储位，但尚未进入实际运行时判断。

### 3.2 当前实现的核心问题

尽管已有组织字段和专业化路由，当前实现仍然停留在“通用四角色流水线”阶段，主要问题如下：

1. `agent.role` 仍然只有 `pm / supervisor / worker / reviewer` 四种粗粒度角色，无法表达“开发主管”和“测试主管”这类主管域职责。
2. 编排器在主管阶段只会查找任意 `supervisor`，不会根据任务域路由到对应主管。
3. `reports_to` 目前只用于查询展示，不参与编排决策。
4. `managed_roles`、`allowed_tools`、`risk_level` 在设计文档中存在，但当前实现中没有形成稳定的运行时模型。
5. 评审失败虽然会把任务状态转为 `revision_required`，但没有“回到哪个主管、由谁重新安排”的闭环。
6. 任务本身没有显式“主管域 / 执行域”字段，当前仍依赖标题关键词去推断 worker specialization。

### 3.3 WP2-07 要解决到什么程度

`WP2-07` 不追求一次实现文档 002 中全部角色，而是先把“主管负责制”真正落地到编排主链路。

本工作包的完成标准不是“角色名更多了”，而是：

1. 任务可以先路由到正确主管，而不是直接落到通用主管。
2. 主管能根据自己管理的角色去分派执行者。
3. 评审打回后，任务能重新回到对应主管进行重分派。
4. 升级条件能从契约进入运行态，而不是停留在 JSON 存储。

## 4. 设计原则

### 4.1 保持向后兼容，避免一次性重写

第二阶段不应该为了组织模型把全部运行时推倒重来。因此本次设计采用“保留粗粒度 role，新增细粒度 role_code”的方式推进。

### 4.2 区分运行时大类和业务职责码

建议把 Agent 身份拆为两层：

1. **运行时大类**：继续保留现有 `role`，用于兼容当前 registry、handler、workflow step 以及运行模式。
2. **业务职责码**：新增 `role_code`，用于表达真实组织职责，例如 `DEVELOPMENT_SUPERVISOR`、`BACKEND_DEV_WORKER`。

### 4.3 主管路由优先于执行者路由

任务应先确定“由哪个主管域负责”，再由主管决定分配给哪个执行者，而不是先做 worker specialization，再反推主管归属。

### 4.4 返工必须回到责任主管

评审结论只能改变任务状态，不能直接决定下一轮执行者。`needs_revision` 的下一步动作必须回到责任主管链路，由主管重新安排。

### 4.5 升级策略必须可执行

`task_contract.escalation_policy` 不应只作为存档字段存在，应进入运行时判断，至少覆盖“输入不足、需高风险工具、输出不满足验收标准”三类场景。

## 5. 最小角色矩阵

为降低实现复杂度，`WP2-07` 建议先落最小角色集，而不是一次引入产品、架构、发布全部主管。

### 5.1 最小角色集

| 层级 | 运行时 role | role_code | 上级 | 主要职责 |
|------|-------------|-----------|------|----------|
| 项目负责人 | `pm` | `PM_SUPERVISOR` | Human | 项目级规划、阶段划分、主管协调、升级汇总 |
| 主管 | `supervisor` | `DEVELOPMENT_SUPERVISOR` | PM | 管理开发类任务，分派后端/前端执行者 |
| 主管 | `supervisor` | `QA_SUPERVISOR` | PM | 管理评审与测试验证，负责质量门 |
| 执行者 | `worker` | `BACKEND_DEV_WORKER` | Development Supervisor | 实现后端逻辑与交付物 |
| 执行者 | `worker` | `FRONTEND_DEV_WORKER` | Development Supervisor | 实现前端页面与交付物 |
| 执行者 | `worker` | `QA_WORKER` | QA Supervisor | 执行测试验证与测试工件产出 |
| 评审者 | `reviewer` | `REVIEWER_WORKER` | QA Supervisor | 独立评审执行结果，给出结论 |

### 5.2 延后角色

以下角色保留到后续工作包，不纳入 `WP2-07` 首批实现：

1. `PRODUCT_SUPERVISOR`
2. `ARCHITECTURE_SUPERVISOR`
3. `RELEASE_SUPERVISOR`
4. `PRD_WORKER`
5. `API_DESIGN_WORKER`
6. `DATA_MODEL_WORKER`
7. `INTEGRATION_TEST_WORKER`
8. `DEPLOY_WORKER`

原因不是这些角色不重要，而是当前最小闭环只需要先把开发主管和测试主管真正接入编排主链路。

## 6. 角色字段设计

### 6.1 Agent 模型建议调整

当前 `Agent` 模型字段不足以表达主管链路，建议新增或显式化以下字段：

| 字段 | 类型 | 建议 | 用途 |
|------|------|------|------|
| `role` | 现有 | 保留 | 运行时大类，兼容现有代码 |
| `role_code` | 新增 `text` | 必须 | 业务职责码，表达真实组织角色 |
| `managed_roles` | 新增 `jsonb` | supervisor 必须 | 表示该主管可管理的下级 role_code 列表 |
| `allowed_tools` | 新增 `jsonb` | 可选 | 表达默认工具模板 |
| `risk_level` | 新增 `text` | 可选 | 表达主管或执行者默认风险级别 |
| `reports_to` | 现有 | 保留 | 组织树与汇报链路 |
| `specialization` | 现有 | 保留 | worker 的专业方向 |

### 6.2 兼容性建议

为了避免打爆当前代码，建议本工作包遵循以下兼容策略：

1. `role` 暂不废弃，继续用于 registry 和 workflow step 的大类路由。
2. 新增的 `role_code` 只用于主管域路由、展示和权限解释。
3. `managed_roles` 不要求第一版就驱动所有权限，只先用于主管分派合法性校验。
4. `allowed_tools` 可以先作为模板字段，后续再与工具权限系统深度整合。

### 6.3 建议的 role_code 常量

建议在 `backend/internal/model/agent.go` 中新增：

1. `PM_SUPERVISOR`
2. `DEVELOPMENT_SUPERVISOR`
3. `QA_SUPERVISOR`
4. `BACKEND_DEV_WORKER`
5. `FRONTEND_DEV_WORKER`
6. `QA_WORKER`
7. `REVIEWER_WORKER`

## 7. 任务域与主管路由设计

### 7.1 当前问题

当前系统只在 `runWorker()` 阶段通过标题关键词推断 specialization，这只能解决“选哪个 worker”的问题，无法解决“先找哪个主管”的问题。

### 7.2 建议新增任务域字段

建议在 `task` 上新增一个显式字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `execution_domain` | `text` | 标识任务归属域，如 `development`、`qa`、`general` |

第一版建议支持以下取值：

1. `development`
2. `qa`
3. `general`

后续再扩展到 `product`、`architecture`、`release`。

### 7.3 任务域来源

`execution_domain` 的来源建议按优先级处理：

1. PM Agent 在任务输出中显式给出。
2. 若未显式给出，则由服务端 fallback 规则根据标题和描述推断。
3. 若仍无法确定，则设为 `general` 并升级给 PM 或人工确认。

### 7.4 主管路由规则

建议新增主管路由函数，其核心逻辑如下：

1. `development` -> `DEVELOPMENT_SUPERVISOR`
2. `qa` -> `QA_SUPERVISOR`
3. `general` -> 默认回退给 `PM_SUPERVISOR` 或升级

主管路由不应再使用“找到任意 supervisor 即可”的策略。

## 8. 编排链路调整建议

### 8.1 当前链路

当前主链路实际是：

`PM -> 任意 Supervisor -> 专业化 Worker -> 任意 Reviewer`

这条链路的主要问题是：

1. 主管不区分部门域。
2. Reviewer 不区分质量门归属。
3. 返工不会回到责任主管。

### 8.2 目标链路

`PM -> 责任主管 -> 执行者 -> QA/Reviewer -> 责任主管 -> PM`

### 8.3 建议的执行流程

#### 阶段 1：PM 分解

PM 负责：

1. 生成阶段和任务。
2. 为每个任务给出 `execution_domain`。
3. 说明是否需要独立评审。

#### 阶段 2：主管规划

主管负责：

1. 接收本域任务。
2. 创建契约。
3. 选择执行者。
4. 写入分派记录。
5. 判断是否需要升级。

#### 阶段 3：执行者交付

执行者负责：

1. 按契约产出工件。
2. 填写交接信息。
3. 报告风险和待决问题。

#### 阶段 4：评审与质量门

评审者负责：

1. 独立检查工件。
2. 给出 `approved / needs_revision` 结论。
3. 评审结果写入 `review_report`。

#### 阶段 5：主管收口

责任主管负责：

1. 对 `approved` 结果进行收口。
2. 对 `needs_revision` 结果重新安排执行者。
3. 对高风险或跨域阻塞进行升级。

### 8.4 返工闭环规则

当评审结论为 `needs_revision` 或 `rejected` 时：

1. `ReviewReportService` 继续负责把任务状态改为 `revision_required`。
2. 编排层必须能够根据任务记录找到责任主管。
3. 责任主管收到返工事件后，重新进入“主管规划 -> 执行者交付”子链路。
4. 执行者不能直接在 `revision_required` 后自行恢复闭环。

## 9. 主管链路所需最小字段变更

为支持上述模型，建议 `WP2-07` 至少落以下字段：

### 9.1 Agent 表

1. `role_code text not null`
2. `managed_roles jsonb not null default '[]'`
3. `allowed_tools jsonb not null default '[]'`
4. `risk_level text not null default 'medium'`

### 9.2 Task 表

1. `execution_domain text not null default 'general'`
2. `owner_supervisor_id uuid null`

### 9.3 TaskAssignment 表

不强制新增列，但建议先通过 `metadata` 补充：

1. `assignment_source: supervisor|review_rework|pm`
2. `supervisor_role_code`
3. `supervisor_id`

### 9.4 TaskContract 表

不新增字段，但需要明确 `escalation_policy` 的推荐结构，例如：

```json
{
  "supervisor_role_code": "DEVELOPMENT_SUPERVISOR",
  "must_escalate_when": [
    "missing_input_artifact",
    "need_unapproved_tool",
    "review_failed_twice"
  ]
}
```

## 10. API 与前端影响面

以下接口需要感知新字段：

1. `POST /api/v1/agents`
2. `PUT /api/v1/agents/{id}`
3. `GET /api/v1/agents`
4. `GET /api/v1/agents/{id}`
5. `GET /api/v1/agents/org-tree`

### 10.2 Task / Workflow 展示

前端至少应能展示：

1. 任务归属域 `execution_domain`
2. 责任主管 `owner_supervisor_id`
3. Agent 的 `role_code`
4. 主管和执行者之间的汇报链

### 10.3 向后兼容要求

第一版不得破坏现有 Demo 和现有 CRUD 接口的基本可用性。未提供新字段时，系统应有默认回退策略。

## 11. 自动化测试建议

`WP2-07` 至少应新增以下测试：

1. Agent `role_code` / `managed_roles` 的 model 与 repository 测试。
2. 主管域路由函数测试。

4. `needs_revision` 后任务重新回到责任主管的链路测试。
5. `escalation_policy` 命中时的升级测试。

建议最小场景集：

1. 开发任务 -> Development Supervisor -> Backend Worker -> Reviewer -> 完成
2. 前端任务 -> Development Supervisor -> Frontend Worker -> Reviewer -> 完成
3. 评审打回 -> 回到 Development Supervisor -> 重新分派 -> 再评审
4. QA 类任务 -> QA Supervisor -> QA Worker
5. 契约命中升级条件 -> 进入升级路径

## 13. 验收清单

`WP2-07` 完成时，至少应满足以下条件：

1. Agent 数据中可明确区分 `role` 与 `role_code`。
2. 主管对象不再是任意 `supervisor`，而是按任务域路由。
3. 至少开发主管和测试主管两条链路真实生效。
4. 评审打回后，任务会回到责任主管而不是停在执行者侧。
5. 升级条件至少有一类能够被系统识别并触发。
6. 控制台中能看到组织链路与主管归属信息。

---

## 14. 结论

`WP2-07` 的本质不是“再加几个 Agent 名字”，而是把当前已经存在的数据模型、组织树字段和能力标签真正接进编排主链路。

如果这一工作包做实，第二阶段后续的 `WP2-08`、`WP2-09`、`WP2-10`、`WP2-11` 都会建立在更稳定的组织模型之上；如果这一工作包继续停留在表面命名层，后续所有工作都会重复建立在“通用 supervisor / worker”假设之上。

因此，本文建议把 `role_code + execution_domain + owner_supervisor_id + rework-to-supervisor` 视为 `WP2-07` 的最小闭环，而不是可选增强项。
