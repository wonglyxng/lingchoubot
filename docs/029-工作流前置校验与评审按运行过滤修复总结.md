# 029-工作流前置校验与评审按运行过滤修复总结

## 变更摘要

本次修复聚焦三个问题：

1. 工作流缺少 reviewer 等关键角色时仍允许启动
2. 单个步骤失败后 workflow_run 仍可能被错误标记为 completed
3. 评审页展示全局历史 review_report，无法区分当前运行与历史运行

本次实现后：

- 启动工作流前会校验活跃 `pm`、`supervisor`、`worker`、`reviewer` 是否齐备，缺失时直接返回 `PRECHECK_FAILED`，不创建 run
- 本地编排与 Temporal 编排均改为步骤失败即将 run 标记为 failed，并停止继续完成后续 run 收尾
- `review_report` 新增 `run_id`，支持按工作流运行过滤；前端评审页支持 `run_id` 过滤，工作流页可跳转查看当前运行的评审记录

## 关键实现说明

### 1. 工作流启动前置校验

新增 `validateWorkflowStartPreconditions`，统一校验项目存在且活跃 `pm`、`supervisor`、`worker`、`reviewer` 角色齐备。

- 本地引擎 `Engine.Run` / `RunAsync` 在创建 run 前执行校验
- `TemporalEngine.RunAsync` 同样在创建 run 前执行校验
- handler 识别 `ErrWorkflowPrecheckFailed` 并返回 `409 PRECHECK_FAILED`

### 2. 失败语义收口

本地编排原先在 task chain 失败后只写日志继续跑，最终仍可能写入 `workflow.completed`。

现在改为：

- `runPhase` 内一旦某个 task chain 失败，立即返回 error
- `Run` / `RunAsync` 接收到阶段错误后调用 `failRun`
- Temporal workflow 在 `Supervisor` / `Worker` / `Reviewer` / `CheckRework` / `CompleteRun` 任一步骤出错时都调用 `ActivityFailRun` 并返回 error

### 3. 评审报告绑定运行

`review_report` 新增可空字段 `run_id`：

- 编排引擎在 `processReviewActions` 中写入当前 run ID
- repository / handler / API 全链路支持 `run_id` 查询过滤
- 前端评审页支持 `?run_id=...` 过滤
- 工作流页增加“查看本次评审”入口，跳转到过滤后的评审页

## 已知限制

1. 启动前校验目前只校验四类基础角色是否存在，不会在启动前精确验证每个任务将来是否一定能匹配到特定 specialization 的 worker
2. 评审页默认仍保留“全局历史视图”，只是新增了 `run_id` 过滤与工作流页跳转入口
3. 历史旧数据中的 review_report 没有 `run_id`，会在评审页显示为空，需要新增运行后才会带上该字段
