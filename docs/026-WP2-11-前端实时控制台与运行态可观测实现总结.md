# WP2-11 前端实时控制台与运行态可观测实现总结

## 1. 变更摘要

将控制台从纯轮询模式升级为 **SSE 优先、轮询兜底** 的实时更新架构，并新增运行态概览仪表盘。

**核心链路**：后端审计事件 → EventHub 内存广播 → SSE Handler 推送 → 前端 useEventStream Hook → 页面自动刷新

## 3. 关键实现说明

### 3.1 EventHub（内存 pub/sub）

- 基于 Go `sync.RWMutex` + `chan` 实现，无外部依赖
- 支持按 topic 过滤订阅（workflow / approval / tool_call / audit）
- 非阻塞发送：慢消费者的事件会被丢弃，SSE 断开后自动重连恢复
- 64 条事件缓冲区

### 3.2 SSE Handler

- 端点：`GET /api/v1/events/stream?topics=workflow,approval`
- 使用 `http.Flusher` 流式推送
- 使用 `http.ResponseController` 禁用写超时（避免 15s WriteTimeout 断连）
- 15 秒心跳保活
- SSE 格式：`id: {event_id}\nevent: {topic}\ndata: {json}\n\n`
- 支持 nginx 代理（`X-Accel-Buffering: no`）

### 3.3 事件集成策略

采用 **审计日志驱动** 策略：所有关键操作均通过 `AuditService.LogEvent()` 记录审计日志，在写入数据库后自动广播 SSE 事件。事件 topic 由 event_type 前缀映射：
- `workflow.*` → topic `workflow`
- `approval*` → topic `approval`
- `tool_call*` → topic `tool_call`
- 其他 → topic `audit`

### 3.4 前端 useEventStream Hook

- 使用标准 `EventSource` API 连接 SSE
- 3 次重连失败后自动降级为轮询模式
- 降级后仍在 30 秒后尝试恢复 SSE
- 返回 `{ connected, mode }` 用于 UI 状态显示
- 回调采用 ref 模式避免不必要的重连

### 3.5 首页运行态仪表盘

新增三个实时面板：
1. **运行中工作流** — 显示 running 状态的工作流列表
2. **待审批** — 显示 pending 状态的审批请求
3. **最近失败** — 显示 failed 状态的工作流

新增最近事件流（最新 8 条审计日志）。

### 3.6 连接状态指示器

所有实时页面（workflows、approvals、tool-calls、audit、首页）均显示：
- 🟢 **实时** — SSE 连接正常
- 🟡 **轮询** — SSE 不可用，已降级为定时轮询

## 6. 验收标准对照

| 验收标准 | 状态 |
|---------|------|
| 工作流状态变化可在前端实时更新，无需刷新页面 | ✅ SSE 推送触发自动刷新 |
| 审批完成、工具调用完成、关键审计事件可实时出现 | ✅ 所有 audit 事件均触发 SSE 广播 |
| SSE 断开后系统能自动降级回轮询，不影响主流程操作 | ✅ 3 次失败后降级，30s 后重试恢复 |
| 控制台可直接演示完整过程 | ✅ 首页仪表盘 + 实时状态指示器 |

## 7. 已知限制

1. EventHub 为进程内内存实现，多实例部署需引入 Redis pub/sub 或 NATS（属 P3 范围）
2. SSE 端点未做鉴权过滤（当前 MVP 整体依赖 API Key 中间件）
3. 工作流、审计、工具调用页已在后续收口中改为增量更新或静默刷新；其余页面仍可继续按实际负载做更细粒度优化
4. EventSource API 不支持自定义 Header，SSE 端点的鉴权走 query param 或 cookie（当前未实现，走全站 Auth 中间件）
