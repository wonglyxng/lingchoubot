## 2. 文档目标

将抽象架构转化为可执行开发计划：

1. 明确 MVP 范围，避免过度扩张
2. 将系统拆分为可并行、可验收、可回滚的模块
3. 提供清晰的实现顺序
4. 从第 1 周开始就能持续产出可验证结果

---

## 3. MVP 定义

灵筹 MVP 的目标是做成一个能稳定跑通下述流程的最小系统：

> 用户输入一个软件项目需求
> → 项目负责人 Agent 理解并拆解
> → 主管 Agent 协同
> → 系统生成任务、工件、审批记录
> → 至少完成一个小型项目的"需求 -> 方案 -> 代码 -> 测试 -> 发布候选包"闭环

---

## 4. MVP 范围

### 4.1 必做范围

1. 项目创建与状态管理
2. Agent 注册、角色定义、上下级关系
3. 任务创建、任务契约定义
4. 任务分派与状态流转
5. 执行 Agent 接收任务并产生工件
6. 独立评审或主管验收打回机制
7. 工件存储与版本记录
8. 交接快照记录
9. 审批请求与审批结果记录
10. 审计日志
11. 简版 Web 控制台
12. 至少一条端到端流程跑通

### 4.2 不纳入 MVP 的范围

- 多租户企业版
- 复杂 RBAC / SSO
- 完整聊天式会话中心
- 高级知识库与向量检索
- 自动生产发布
- 复杂绩效系统
- NATS 事件总线
- Temporal 深度接入
- 高拟真 Agent 社会模拟

---

## 5. MVP 成功标准

### 5.1 功能闭环

- 创建项目并指定项目负责人 Agent
- 生成阶段与任务树
- 分派至少 2 类主管 Agent
- 执行至少 2 类执行 Agent
- 生成至少 3 个工件
- 至少 1 个任务经历"执行 -> 评审 -> 打回或通过"
- 至少 1 次审批记录
- 输出项目交付总结

### 5.2 稳定性

- 任务状态机无明显错乱
- 核心数据不丢失
- 工件和任务可追溯
- 审计日志能回放关键步骤

### 5.3 工程质量

- 代码仓结构清晰
- 模块边界明确
- 数据库 migration 可重复执行
- 本地部署可一键跑起

### 5.4 可演进

- 能新增 Agent 角色而不重写控制平面
- 能新增工具而不重写任务模型

### 5.5 可演示

在 Web 控制台中能看到：项目、阶段、任务树、Agent 组织树、工件列表、审批中心、审计时间线。

## 6. 总周期与节奏

建议将 MVP 划分为 **10 周**，包含 1 周缓冲。

每周节奏：
- 周初：确认本周范围
- 周中：实现与联调
- 周末：Demo + 缺陷清理 + 下周计划

## 8. 分周详细计划

### 第 1 周：基础骨架

**目标：** 搭建最小工程骨架，确保仓库、数据库、API 进程、前端壳、Docker 环境都能启动。

**交付物：**
1. Monorepo 初始化（Go backend + Next.js frontend）
2. PostgreSQL 本地开发环境（Docker Compose）
3. Migration 框架接入（golang-migrate）
4. 基础配置管理（.env）
5. 健康检查接口（`/healthz`、`/readyz`）
6. 基础日志与错误处理中间件
7. README（本地启动指南）

**验收标准：**
- `docker compose up` 后核心服务可启动
- 能执行 migration
- 能访问健康检查接口
- Web 页面可正常显示基础导航壳

**简化策略：** 暂不接入 NATS、Temporal、MinIO。所有配置先采用本地 `.env`。

---

### 第 2 周：核心表与 CRUD

**目标：** 实现项目、阶段、Agent、任务四个核心对象的最小可用 CRUD 和状态变更。

**交付物：**
1. 核心表 migration（project、project_phase、agent、task、audit_log）
2. Repository 层
3. project / phase / agent / task CRUD API
4. 基本状态流转规则
5. 基础审计日志写入

**验收标准：**
- 能通过 API 创建项目、阶段、Agent、任务
- 能更新任务状态
- 审计表中可见对应记录

**简化策略：** 先不做复杂权限判断，先保证"可创建、可查询、可变更"。

---

### 第 3 周：任务契约与分派

**目标：** 让系统具备任务契约定义和组织化分派能力。

**交付物：**
1. task_contract 表与 API
2. task_assignment 表与 API
3. Agent 组织树查询（reports_to 关系）
4. 交接快照结构（handoff_snapshot 表）
5. 任务分派历史记录

**验收标准：**
- 能为任务定义契约（范围、非目标、完成定义、验证步骤）
- 能将任务分派给指定 Agent
- 能查询 Agent 组织树
- task_assignment 有完整分派历史

**简化策略：** 先用同步调用，暂不引入消息队列。

---

### 第 4 周：工件与版本

**目标：** 建立工件驱动推进的骨架，让任务有真正可追踪的交付物。

**交付物：**
1. artifact 与 artifact_version 表与 API
2. 工件注册逻辑（元数据入库 + URI 指向存储）
3. 工件版本递增逻辑
4. review_report 表与 API
5. MinIO 接入（工件文件存储）

**验收标准：**
- 执行 Agent 能提交工件
- 工件版本可递增
- 能创建独立评审报告
- 评审失败时能打回 `revision_required`

**简化策略：** 先支持 URI + 元数据入库，文件存储接 MinIO。

---

### 第 5 周：审批与审计

**目标：** 完成审批闭环和审计时间线。

**交付物：**
1. approval_request 表与 API
2. 审批状态流转（pending -> approved / rejected）
3. 审批与任务/工件关联
4. 审计时间线聚合查询 API
5. 审计日志补全（覆盖所有关键动作）

**验收标准：**
- 能发起审批请求
- 用户能通过 API 审批/拒绝
- 审批结果能回写任务状态
- 审计时间线可按项目/任务筛选

**简化策略：** 先做单步骤审批，不做复杂多级审批流。

---

### 第 6 周：Agent Runtime

**目标：** 让系统第一次具备 Agent 执行能力，跑通最小 Supervisor -> Worker 链路。

**交付物：**
1. Agent Runtime 协议定义（输入/输出 schema）
2. Mock Agent 执行器（不接 LLM，用固定逻辑模拟）
3. Supervisor -> Worker -> Reviewer 最小链路
4. 基于交接快照的恢复信息记录
5. 最小工作流编排器（应用内状态推进）

**验收标准：**
- 项目负责人 Agent 能下发任务给主管
- 主管 Agent 能下发任务给执行 Agent
- 执行 Agent 能回传结构化结果
- 能看到任务契约与交接信息
- 能看到独立评审结论

**简化策略：** Agent 先用 Mock 实现，重点把协议和分派逻辑定下来。后续再接 LLM。

---

### 第 7 周：工具网关

**目标：** 让执行 Agent 可以通过受控工具网关执行有限动作。

**交付物：**
1. Tool Gateway 抽象（统一 request/response 结构）
2. tool_call 记录表与 API
3. 首批工具接入（doc_generator、artifact_storage、test_runner）
4. 工具权限校验
5. 工具调用日志

**验收标准：**
- Agent 可通过 Tool Gateway 调用至少 2 个工具
- 每次调用都有 tool_call 记录
- 未授权调用会被拒绝
- 失败结果能反馈回任务状态

**简化策略：** Shell 仅允许白名单命令，Git 先只做只读。所有工具统一 request/response 结构。

---

### 第 8 周：Web 控制台

**目标：** 将核心能力可视化，使灵筹具备最小可演示的管理台。

**交付物：**
1. 项目列表页
2. 项目详情页（含阶段、最近任务、工件、待审批项）
3. 任务看板页（按状态分栏）
4. Agent 组织树页
5. 审批中心页
6. 审计时间线页

**验收标准：**
- 用户能从 Web 页面完成核心浏览操作
- 能看到项目、任务、工件、审批、审计五大核心对象
- 控制台足以支撑 Demo

**简化策略：** UI 可以朴素，但信息结构要清晰。先保证真实数据流，不做复杂动画。

---

### 第 9 周：端到端联调

**目标：** 用一个真实小项目跑通 MVP 完整闭环。

**交付物：**
1. 真实 Demo 项目一套
2. Demo 跑通脚本
3. 缺陷清单与修复
4. 关键路径集成测试

**推荐 Demo 场景：**
"根据需求文档自动拆解并生成前后端骨架 + 测试计划 + 发布候选包"

**验收演示流程：**
1. 用户创建项目
2. 项目负责人 Agent 输出阶段计划
3. 主管与执行侧明确任务契约
4. 执行 Agent 生成工件
5. 执行 Agent 提交交接快照
6. 测试主管给出独立评审结论
7. 如未通过则打回并修正
8. 发布主管发起审批
9. 用户审批
10. 系统输出最终交付总结

**验收标准：**
- 至少一个完整 Demo 跑通
- 不依赖手工篡改数据库
- 所有关键对象能在 Web 控制台看到
- 至少一条审计时间线可完整回放

---

### 第 10 周：缓冲与收尾

4. 技术债清单
5. 下一阶段待办列表

## 10. 推荐开发顺序

### 第一步：先搭数据骨架
先生成 migrations、schema、repository 接口、基础实体定义。不要一上来先写 Agent Prompt 和工作流。

### 第二步：做控制平面
先完成 project service、task service、agent registry、artifact service、approval service、audit service。Agent 执行先用最小 mock。

### 第三步：固定协议，再接 Agent Runtime
先把 task input/output schema、task contract schema、tool call schema、approval schema、artifact schema 定死，之后再接入 LLM 驱动的 Agent。

### 第四步：先跑同步链路，再异步化
先跑通 API 创建任务 -> 调用 mock agent -> 返回结果 -> 更新状态 -> 产生工件。再逐步接入 queue、workflow、event bus。

### 第五步：Web 只做真实数据
前端页面必须优先消费真实 API，不做静态假数据页面。

## 15. 风险与应对

| 风险 | 表现 | 应对 |
|------|------|------|
| 范围蔓延 | 想把聊天、知识库、多租户一起做 | 严格按 MVP 范围，新功能必须回答"是否影响核心闭环" |
| Agent 设计过虚 | 花大量时间写 Prompt，没有稳定协议 | 先建任务系统 + 工件系统 + 审批系统，再接 Agent |
| 工作流过早复杂化 | 在 Temporal 细节上花太久 | 先做最小可恢复编排，用 adapter + mock 过渡 |
| 工具权限失控 | 给执行 Agent 太多 shell/git 权限 | 默认最小权限，先只开文档生成、工件写入、测试运行 |
| 前端抢进度 | 页面很多但后端未稳定 | 第 8 周前端集中推进，之前只做壳和基础联调 |
| 代码扩散 | 一次性生成过多文件 | 每次只实现一个明确模块，不越界修改无关模块 |
