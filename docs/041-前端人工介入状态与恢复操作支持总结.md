# 041-前端人工介入状态与恢复操作支持总结

## 变更摘要

本次补齐了前端对工作流 `waiting_manual_intervention` 状态的支持，使控制台能够识别真实 LLM 失败后的人工介入挂起状态，并在工作流页提供恢复执行入口。

同时对齐了前端工作流类型定义与后端返回结构，修复了页面仍依赖旧字段 `result_summary`、`step_name`、`duration_ms` 的问题。

## 文件清单

- `frontend/src/lib/types.ts`
- `frontend/src/lib/api.ts`
- `frontend/src/lib/utils.ts`
- `frontend/src/app/workflows/page.tsx`
- `frontend/src/app/page.tsx`
- `docs/041-前端人工介入状态与恢复操作支持总结.md`

## 关键实现说明

### 1. 工作流类型与接口对齐

- `WorkflowRun` 使用后端实际返回的 `summary` 字段，不再使用前端遗留的 `result_summary`
- `WorkflowStep` 使用 `name`、`agent_role`、`task_id`、`phase_id`、`started_at`、`completed_at`
- 前端 API 客户端新增 `workflows.resume(id)`，对应后端 `POST /api/v1/orchestrator/runs/{id}/resume`

### 2. 工作流页支持人工介入恢复

- 工作流卡片支持展示 `waiting_manual_intervention`
- 当运行处于 `waiting_manual_intervention` 或 `waiting_approval` 时，显示“恢复执行”按钮
- 恢复成功后立即刷新当前运行详情和步骤列表
- 运行详情区域增加人工介入说明，明确这是由真实 LLM 调用失败导致的暂停，而不是普通失败终态

### 3. 首页增加待人工介入视图

- 首页不再展示“最近失败”列表，而是展示“待人工介入”列表
- 该列表直接拉取 `waiting_manual_intervention` 状态的工作流运行，便于值守和人工恢复

### 4. 假设与范围

- 假设后端已部署包含 `waiting_manual_intervention` 状态和 `resume` 接口的版本
- 本次仅补齐首页与工作流页，不扩散修改到其他页面
- 保持现有 SSE 轮询模式，不引入新的前端状态管理层

## 已知限制

- 当前首页只展示待人工介入运行，不直接在首页触发恢复，恢复操作仍在工作流页执行
- `audit` 和 `tool-calls` 页面仍存在历史遗留的未使用参数 lint warning，本次未顺手修改无关页面
- 项目详情页暂未增加待人工介入运行的专门展示入口
