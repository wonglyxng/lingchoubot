# WP2-09 真实 LLM 质量治理与执行护栏 — 实现总结

## 1. 变更摘要

实现 WP2-09 工作包，目标是将已接入的真实 LLM 从"能调用模型"提升为"可控、可测、可回归"的执行能力。主要包含以下能力：

1. **提示词版本管理** — 为每个角色的 system prompt 建立版本号和变更记录，可追踪提示词演进
2. **LLM 输出 schema 校验** — 所有角色的 LLM 输出经过结构化校验，不合格时不会静默写入错误结果
3. **LLM 调用审计增强** — `ChatJSONWithMeta` 返回元数据（model、耗时、token 用量、错误原因），为成本和质量分析提供数据基础
4. **失败降级策略** — LLM 调用失败、JSON 解析失败、输出校验失败时，可自动回退到 MockRunner，确保主链路可用
5. **固定评测样本集** — 为 PM / Supervisor / Worker / Reviewer 各提供一个标准评测输入，用于模型升级前后的回归对比

## 3. 关键实现说明

### 3.1 提示词版本管理

```go
var promptVersions = map[string]PromptVersion{
    "pm":         {Role: "pm", Version: "1.0.0", ChangeLog: "初始版本：项目分解为阶段+任务"},
    "supervisor": {Role: "supervisor", Version: "1.0.0", ChangeLog: "初始版本：任务契约+分派+状态流转"},
    "worker":     {Role: "worker", Version: "1.0.0", ChangeLog: "初始版本：执行任务产出工件+交接"},
    "reviewer":   {Role: "reviewer", Version: "1.0.0", ChangeLog: "初始版本：独立评审+结论"},
}
```

版本号随 prompt 内容变更递增，方便追踪哪个版本的 prompt 产生了哪个质量的输出。

### 3.2 LLMCallMeta — 审计元数据

```go
type LLMCallMeta struct {
    Model            string
    DurationMs       int64
    PromptTokens     int
    CompletionTokens int
    TotalTokens      int
    StatusCode       int
    Error            string
    PromptVersion    string
}
```

每次 LLM 调用都返回 meta，Runner 在日志中输出 `duration_ms`、`tokens`、`prompt_version` 等关键指标。后续可接入 `tool_call` 或 `workflow_step` 的 metadata 字段做持久化。

### 3.3 输出 Schema 校验

| 角色 | 校验规则 |
|------|---------|
| PM | 必须有 ≥1 个 phase 和 ≥1 个 task，phase.name 不为空，task.title 和 phase_name 不为空 |
| Supervisor | 必须有 ≥1 个 contract，scope 不为空，done_definition ≥2 条，acceptance_criteria ≥1 条 |
| Worker | 必须有 ≥1 个 artifact，name 和 artifact_type 不为空 |
| Reviewer | 必须有 ≥1 个 review，verdict 必须是 approved 或 needs_revision，findings ≥1 条 |

### 3.4 三级降级链路

```
LLM 调用
  ├─ 成功 → JSON 解析
  │          ├─ 成功 → Schema 校验
  │          │          ├─ 通过 → 返回输出 ✓
  │          │          └─ 失败 → fallback → MockRunner → [降级] 标注
  │          └─ 失败 → fallback → MockRunner → [降级] 标注
  └─ 失败 → fallback → MockRunner → [降级] 标注
```

降级时输出的 summary 会被添加 `[降级]` 前缀，方便在审计和 UI 中识别。

### 3.5 固定评测样本

4 个样本覆盖：项目分解、契约创建、后端代码执行、代码评审。输入固定、可重复运行。同时有测试验证 MockRunner 能对这些样本产出通过 schema 校验的输出。

## 4. 运行方式

```bash
# 启动后端（启用 LLM + 降级）
LLM_ENABLED=true LLM_API_KEY=sk-xxx LLM_FALLBACK_ENABLED=true go run cmd/api/main.go

# 禁用降级（LLM 失败直接报错）
LLM_ENABLED=true LLM_API_KEY=sk-xxx LLM_FALLBACK_ENABLED=false go run cmd/api/main.go

# Mock 模式（不使用 LLM）
go run cmd/api/main.go
```

## 6. 已知限制

1. **LLMCallMeta 尚未持久化** — 当前只输出到日志，未写入 `tool_call` 或 `workflow_step.metadata`，后续 WP 可接入
2. **提示词版本未存储到数据库** — 版本信息在代码中维护，未做数据库持久化和历史查询
3. **评测样本只有 4 个** — 每角色 1 个最小样本，后续应扩展为多场景覆盖
4. **校验规则较宽松** — 只检查结构完整性，未检查语义合理性（如 phase 是否有序、task 是否归属正确 phase）
5. **降级标注是字符串前缀** — `[降级]` 标注只修改 summary，未作为独立字段暴露给审计
6. **token 用量依赖 API 返回** — 如果 LLM API 不返回 usage 字段（某些兼容 API），token 数据为 0
