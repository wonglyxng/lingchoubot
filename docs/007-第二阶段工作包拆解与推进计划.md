## 2. 目标

MVP 第一阶段 15 个工作包已经全部完成，当前系统已经具备最小闭环能力。第二阶段的目标不是继续堆功能页面，而是把当前基于内存、同步、Mock 的主链路升级为可持续演进的工程底座。

第二阶段只聚焦四件事：

1. 正式接入 Temporal
2. 正式接入 NATS / Outbox
3. 增加更多执行 Agent
4. 建立更细粒度工具权限

---

## 3. 当前基线判断

基于当前仓库实现，第二阶段规划需要建立在以下现状上：

1. 工作流运行态仍为内存存储，`RunStore` 使用 map + mutex，进程重启后运行记录丢失。
2. 编排器仍是应用内同步线性执行，尚未形成可替换 Temporal 的 workflow adapter。
3. 工具权限基于 Agent 的字符串权限列表和通配符匹配，可用但粒度较粗。
4. Agent Runtime 通过角色到 Runner 的注册表挂接，适合 Mock 阶段，但不适合进一步扩展为多能力执行体系。
5. 当前仍有若干收尾缺口未闭合：MinIO 实际上传未实现、审批 rejected 回路未在 Demo 覆盖、自动化测试缺失、鉴权中间件缺失。

结论：第二阶段应先完成“可迁移、可持久、可控制”的底座改造，再扩展 Agent 数量。

---

## 4. 范围与非目标

### 4.1 第二阶段范围

- 编排运行态持久化
- Temporal 接入与编排迁移
- Outbox 事件建模与 NATS 投递
- Agent 能力模型扩展与新增执行 Agent
- 工具权限细化与鉴权落地
- 阶段 2 集成测试与演示回归

### 4.2 第二阶段非目标

- 向量检索与知识库
- 多租户 / 团队版
- 复杂审批流
- 高级诊断与回放中心
- 复杂项目模板市场

这些内容应继续留在后续阶段，避免范围再次膨胀。

---

## 5. 进入条件

严格来说，阶段 2 开始前应先补完第 10 周收尾项。为了避免阶段切换阻塞，建议将这些事项合并为阶段 2 的前置工作包处理。

前置缺口如下：

1. 审批 rejected -> task revision_required 的 Demo 覆盖
2. MinIO 文件上传从“仅 URI 入库”升级为“真实文件写入 + 元数据入库”
3. 工作流主链路最少补齐自动化测试骨架
4. API 鉴权方案定稿
5. 演示材料、使用说明、技术债清单收口

---

## 6. 工作包拆分

### WP2-00 收尾与准入基线

**目标：** 关闭 MVP 收尾遗留项，为 Temporal / NATS 接入建立稳定基线。

**范围：**
- 审批 rejected 链路补测并补 Demo
- MinIO 实际上传打通
- 编排、审批、工具权限补最少自动化测试
- 梳理 API 鉴权方案
- 更新文档与演示材料

**交付物：**
- MinIO 上传能力可用
- rejected 路径端到端验证通过
- 最少一组单元测试 + 一组集成测试
- 鉴权设计文档

**验收标准：**
- Demo 同时覆盖 approved / rejected 两条审批路径
- 工件文件可真实上传到对象存储
- 核心回归不再只依赖手工 Demo

---

### WP2-01 编排运行态持久化

**目标：** 将当前内存 `RunStore` 升级为可持久化、可恢复的运行态存储。

**范围：**
- 新增 `workflow_run`、`workflow_step` 等持久化模型与 migration
- repository / service 层落地运行态读写
- 编排器查询接口改为基于数据库
- 支持 run 恢复、失败重试、运行历史查询

**交付物：**
- 数据库表与 repository
- 持久化工作流运行查询 API
- 失败恢复基础能力

**验收标准：**
- API 进程重启后仍可查询历史 run
- 至少支持对失败 run 的重新执行或恢复标记
- 审计时间线可关联 run 与 step

**依赖：** WP2-00

---

### WP2-02 Temporal 适配层与首条编排迁移

**目标：** 将当前应用内同步编排器迁移为可切换的 workflow engine 结构，并先迁移一条标准主流程。

**范围：**
- 抽象 workflow engine adapter 接口
- 接入 Temporal client / worker 基础设施
- 将 PM -> Supervisor -> Worker -> Reviewer 主链路迁移为 Temporal Workflow + Activities
- 保留本地内置编排器作为 fallback 或开发模式

**交付物：**
- Temporal 接入配置
- workflow adapter 抽象
- 一条可运行的 Temporal 主流程

**验收标准：**
- 可在本地 docker compose 或开发环境启动 Temporal 相关组件
- 主流程可由 Temporal 驱动完成一次完整执行
- 不破坏现有 HTTP API 的启动和查询方式

**依赖：** WP2-01

---

### WP2-03 Outbox 事件模型与 NATS 接入

**目标：** 把关键状态变更从同步串行调用演进为可投递、可追踪的领域事件流。

**范围：**
- 设计领域事件模型与 outbox 表
- 将任务状态流转、工件创建、审批决策等关键动作写入 outbox
- 接入 NATS / JetStream 或最小 NATS 发布能力
- 增加 outbox relay / publisher 进程

**交付物：**
- outbox migration 与 repository
- 事件发布器
- NATS 基础部署配置

**验收标准：**
- 关键领域动作能稳定写入 outbox
- relay 可将事件投递到 NATS
- 投递失败具备重试与审计记录

**依赖：** WP2-01

---

### WP2-04 Agent 能力模型与新增执行 Agent

**目标：** 从“按角色注册 Mock Runner”升级为“按能力路由 + 多执行 Agent”的运行模型。

**范围：**
- 扩展 Agent 模型，区分角色、能力、工具权限、执行类型
- 将 registry 从单纯 role -> runner 升级为 capability-aware 路由
- 新增至少 2 到 3 个执行 Agent
- 为任务分派增加最小路由规则

**建议新增 Agent：**
- `backend_worker`：后端接口与数据模型实现
- `frontend_worker`：前端页面与交互实现
- `qa_worker`：测试计划、用例、回归验证
- `release_worker`：发布候选包与交付清单整理

**交付物：**
- Agent 能力模型升级
- 新增执行 Agent 实现
- 任务路由规则

**验收标准：**
- 系统可根据任务类型将任务分派到不同执行 Agent
- 至少两类不同执行 Agent 能产出不同类型工件
- Agent 注册与查询 API 能展示能力信息

**依赖：** WP2-00

---

### WP2-05 细粒度工具权限与鉴权闭环

**目标：** 将当前字符串权限匹配升级为“身份鉴权 + 权限范围 + 工具动作级授权”的受控模型。

**范围：**
- API 鉴权中间件落地
- 工具权限拆分为资源、动作、范围三级
- 敏感工具增加审批或白名单限制
- 工具调用审计补全 denied / failed / escalated 场景

**交付物：**
- 鉴权中间件
- 新权限模型及迁移方案
- 工具动作级授权实现

**验收标准：**
- 未鉴权请求无法调用受保护 API
- Agent 即使具备工具访问权，也只能访问被授权动作
- 高风险工具调用可被拒绝并写入审计

**依赖：** WP2-00

---

### WP2-06 阶段 2 集成回归与演示升级

**目标：** 让第二阶段不是“组件上线”，而是“新底座可被端到端证明”。

**范围：**
- 构建覆盖 Temporal / NATS / 多 Agent / 工具权限的 Demo 脚本
- 更新 Web 控制台展示 run、step、event、tool 权限信息
- 输出阶段 2 验收文档与技术债清单

**交付物：**
- 新 Demo 脚本
- 阶段 2 验收报告
- 回归测试结果文档

**验收标准：**
- 一条完整链路可跨多 Agent 执行并带有事件轨迹
- Temporal 执行状态、事件发布结果、工具授权结果可回放
- 演示环境能稳定复现

**依赖：** WP2-02、WP2-03、WP2-04、WP2-05

---

## 7. 推荐执行顺序

建议按以下顺序推进，而不是并行乱开：

1. WP2-00 收尾与准入基线
2. WP2-01 编排运行态持久化
3. WP2-02 Temporal 适配层与首条编排迁移
4. WP2-03 Outbox 事件模型与 NATS 接入
5. WP2-04 Agent 能力模型与新增执行 Agent
6. WP2-05 细粒度工具权限与鉴权闭环
7. WP2-06 阶段 2 集成回归与演示升级

其中 WP2-03、WP2-04、WP2-05 可以在 WP2-02 稳定后有限并行，但不建议在 WP2-01 之前启动。

## 8. 预计影响面

### 8.1 数据库

大概率需要新增以下表或等价结构：

- `workflow_run`
- `workflow_step`
- `outbox_event`
- `agent_capability` 或等价能力字段
- 更细粒度的权限关联表

### 8.2 后端模块

- `backend/internal/orchestrator/`
- `backend/internal/runtime/`
- `backend/internal/gateway/`
- `backend/internal/repository/`
- `backend/internal/service/`
- `backend/internal/handler/`
- `backend/migrations/`

### 8.3 基础设施

- `docker-compose.yml` 需要补充 Temporal 与 NATS 相关服务
- 本地开发配置需要新增 Temporal / NATS / MinIO / 鉴权参数

---

## 9. 阶段 2 验收标准

阶段 2 完成时，至少应满足以下条件：

1. 工作流运行记录持久化，不再依赖内存存活
2. 主流程可由 Temporal 实际驱动执行
3. 关键领域事件可经 outbox 发布到 NATS
4. 至少三类执行 Agent 可参与同一项目链路
5. API 与工具调用具备最小鉴权与细粒度授权
6. 新 Demo 可演示“编排 + 事件 + 多 Agent + 权限 + 审计”完整链路

---

## 10. 不建议现在做的事

以下事项仍不应提前进入第二阶段：

1. 向量知识库与检索召回
2. 多租户与复杂组织权限
3. 高级审批流与多级会签
4. 大量新的前端页面重构
5. 把 Mock Agent 直接替换成真实 LLM 全自动执行

原因很直接：底座还未持久化、事件化、权限化，先扩复杂能力会把调试成本放大。

---

## 11. 建议的下一次执行单位

如果按“一个明确工作包一次推进”的方式继续开发，建议下一次直接启动：

**WP2-00 收尾与准入基线**

优先级排序如下：

1. 审批 rejected 路径补测
2. 自动化测试骨架
3. MinIO 实际上传
4. 鉴权方案定稿
5. 演示材料与技术债文档收口

这个工作包完成后，再进入 WP2-01，会明显降低后续 Temporal / NATS 改造的返工概率。
