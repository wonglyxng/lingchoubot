# 042-SSE列表刷新闪烁修复总结

## 变更摘要

本次修复了前端工作流页面在接入 SSE 后仍表现得像轮询刷新的问题。

根因不是 SSE 没生效，而是页面层叠加了两套刷新机制：

- workflow SSE 事件到达后直接全量重拉运行列表
- 展开中的运行卡片继续每 2 秒拉取详情

同时，轮询兜底路径复用了带全屏 loading 的列表加载方法，导致 SSE 异常降级到轮询时页面列表会周期性闪烁。

## 文件清单

- `frontend/src/app/workflows/page.tsx`

## 关键实现说明

### 1. 工作流页改为事件优先的精准刷新

- 解析 workflow SSE 事件中的 `data.after_state.run_id`
- 优先按 run_id 调用 `GET /api/v1/orchestrator/runs/{id}` 更新单条运行
- 只有无法解析 run_id 或单条拉取失败时，才退回到静默全量刷新列表

这样避免了每个 workflow 事件都触发一次 `list` 全量重拉。

### 2. 轮询兜底不再触发整页 loading 闪烁

- 初次进入页面仍显示 loading
- SSE 失败后的 `onPoll` 改为静默刷新，不再反复切换 `loading=true`

这样即使 SSE 断开进入轮询兜底，也不会出现页面整块一闪一闪的问题。

### 3. 展开卡片只在必要时轮询

- 展开卡片只在未启用 SSE 实时更新时才继续轮询运行详情
- 当 SSE 正常连接时，优先依赖事件驱动的单条运行更新
- 对步骤列表增加等值判断，避免内容未变化时重复 `setState`

### 4. 首页事件刷新做合并触发

- 首页保留 SSE + 轮询兜底结构
- 对高频事件增加 300ms 合并窗口，避免 workflow / audit / tool_call 高频事件导致面板频繁重绘

## 已知限制

- 当前 workflow 事件的顶层 `target_id` 仍是 `project_id`，前端需要从 `data.after_state.run_id` 中提取运行 ID；如果后端后续将 workflow 事件直接标准化为 `run_id`，前端逻辑还能进一步简化
- 展开的运行卡片在 SSE 不可用时仍会保留轮询兜底，以保证步骤进度可见
- `audit` 和 `tool-calls` 页面仍存在历史遗留的未使用参数 lint warning，本次未修改无关页面
