# 036-工件内容落盘与评审审批上下文补全实现总结

## 1. 变更摘要

本次实现补齐了“工件生成 -> 评审 -> 审批 -> 控制台展示”的内容链路，解决以下四类问题：

- 工作流生成的工件正文此前只写数据库元数据，不会进入 MinIO；现在会优先上传到对象存储，并在版本元数据中保留可预览正文。
- reviewer 之前拿不到任务交付物上下文，导致出现“0 个交付物已通过评审”；现在会把任务最新工件版本传入评审输入。
- review_report 与自动生成的 approval_request 之前缺少可直接审批的内容；现在会附带摘要、发现、建议和关联交付物信息。
- 前端工件页、评审页、审批页此前只展示编号或标题；现在会展示正文预览、评审内容和审批依据。

## 3. 关键实现说明

### 3.1 工件正文真正落盘

- 为 `ArtifactService` 增加可选 `ArtifactContentStore` 接口。
- 复用现有 `ArtifactStorageTool`，在 API 进程和 Temporal worker 装配时注入到 `ArtifactService`。
- 当 `ArtifactVersion.Content` 有值时：
  - 优先上传到 MinIO / fallback mock MinIO；
  - 回填 `uri`、`size_bytes`、`checksum`；
  - 在 `metadata` 中写入 `inline_content`、`source_name`、`stored_in`，供控制台预览。

### 3.2 reviewer 基于真实交付物评审

- 在本地编排器与 Temporal reviewer 活动中，评审前会加载任务下的工件及其最新版本。
- 新增 `runtime.ArtifactCtx` 的 `version_id`、`version`、`content_type`、`content` 字段。
- reviewer 输入现在会带上真实 `Artifacts`，从而消除“0 个交付物已通过评审”的假结果。

### 3.3 review / approval 内容补全

- `review_report.metadata` 现在会记录 `artifact_count` 和 `artifacts[]`。
- 自动创建的 `approval_request.metadata` 会补充：
  - `review_id`
  - `review_summary`
  - `findings`
  - `recommendations`
  - `artifact_count`
  - `artifacts[]`
  - 以及既有的 `run_id`、`phase_id`
- 审批描述文案也会带上评审摘要和交付物数量，避免只剩报告编号。

### 3.4 控制台展示升级

- 工件页展示最新版本、存储 URI、来源、正文预览。
- 评审页展示评审摘要、主要发现、整改建议、关联交付物与内容预览。
- 审批页展示审批依据，包括评审摘要、发现、建议和关联交付物预览。

## 4. 运行方式

### 后端

在 `backend` 目录执行：

```bash
go test ./...
```

如需体验真实对象存储链路，请确保通过 `docker-compose.yml` 启动 MinIO，并为后端提供正确的 MinIO 配置。

### 前端

在 `frontend` 目录执行：

```bash
npm run build
```

前端页面会直接消费已有 API：

- `GET /api/v1/artifacts`
- `GET /api/v1/artifacts/{id}/versions`
- `GET /api/v1/reviews`
- `GET /api/v1/approvals`

## 6. 已知限制

- 前端当前展示的是 `metadata.inline_content` 预览，不是直接从 MinIO 拉取原始对象内容。
- 若工件正文很大，控制台当前会做截断展示，仍以预览为主。
- 已有历史数据如果是在本次修复前生成，旧 review / approval 记录不会自动补写上下文，需要重新跑工作流或重新生成评审/审批数据。
