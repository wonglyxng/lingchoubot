# 049-返工工件版本复用与单条评审约束总结

## 1. 变更摘要

本次修复了工作流返工场景下的两个语义错位问题：

1. 同一任务在返工时不再重复创建多个同义工件，而是复用同任务、同名、同类型的 artifact，并追加新的 artifact_version。
2. Reviewer 在单次任务评审中被强制约束为只输出 1 条任务级 review，避免对同一任务重复触发 `pending_approval` 状态流转。

该修复直接解决了“第三轮 reviewer 返回多条 review，导致任务从 `pending_approval` 再次流转到 `pending_approval` 报错”的问题根因。

## 3. 关键实现说明

### 3.1 返工工件改为版本递增

原实现中，`processArtifactActions` 每次收到 Worker 的 artifact 输出都会调用 `CreateWithInitialVersion`，导致同一任务的返工结果不断生成新的 artifact 记录。

现在改为：

- 先按 `task_id + artifact_type` 拉取候选 artifact。
- 若存在“同名且同类型”的 artifact，则调用 `AddVersion` 追加版本。
- 只有在不存在可复用 artifact 时，才创建新的 artifact 和初始版本。

这样可让 PRD、设计稿等返工场景保持“单工件多版本”的语义，Reviewer 看到的将是该工件的最新版本，而不是同义工件列表。

### 3.2 Reviewer 输出改为单条任务级结论

原实现允许 Reviewer 返回任意数量的 `reviews[]`，而编排层会逐条创建 `review_report` 并逐条触发状态流转。这与任务状态机“单次评审产生单次推进”的语义不一致。

现在改为：

- 提示词明确要求 `reviews` 必须且只能包含 1 条任务级评审结论。
- LLM 校验层要求 Reviewer 输出“恰好 1 条 review”。
- 编排层 `processReviewActions` 再做一次防御性检查，发现多条 review 直接报错，不再部分成功、部分失败。

### 3.3 测试基线同步调整

原有集成测试默认“返工一次 = 多一个 artifact”。现在同步更新为：

- 返工后 artifact 数量保持不变。
- version 数量递增。
- 每个返工过的 artifact 都应存在多条 version 记录。

## 4. 运行方式

本地运行方式不变：

1. `docker compose up --build`
2. 创建项目并启动工作流。
3. 在返工场景下观察工件列表与版本列表：同一任务下的同名 PRD 应保持为 1 个 artifact，多轮返工只增长 version。
4. 查看工作流评审步骤：单次 Reviewer 步骤应只产生 1 条 review_report。

## 6. 已知限制

1. 当前 artifact 复用策略基于“同任务、同名、同类型”匹配；如果后续希望支持更稳定的工件身份标识，仍建议引入显式的逻辑工件键。
2. Reviewer 现在被约束为单条任务级结论，适合当前任务状态机；如果未来要支持“同任务多工件独立审批”，需要重新设计 review_report 与状态推进模型。
3. 现有历史数据中如果已经存在多个同义 artifact，本次修复不会自动合并历史记录，只会约束后续新流程按版本递增收敛。
