## 3. 设计原则

- 主键统一使用 UUID（`uuid-ossp` 扩展）
- 时间字段统一使用 `TIMESTAMPTZ`
- 核心状态字段使用 `TEXT + CHECK constraint`
- JSONB 仅用于半结构化元数据，不替代核心关系模型
- 每次 schema 变更必须有对应 migration
- 不允许手工改库而不落 migration

### 5.11 artifact_version

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID, PK | 主键 |
| artifact_id | UUID, FK → artifact ON DELETE CASCADE | 所属工件 |
| version | INT, NOT NULL | 版本号，同一工件内自动递增 |
| uri | TEXT | 存储地址（MinIO/S3 URI 或本地路径） |
| content_type | TEXT | MIME 类型 |
| size_bytes | BIGINT | 文件大小 |
| checksum | TEXT | 校验和 |
| change_summary | TEXT | 本版本变更说明 |
| created_by | UUID, FK → agent ON DELETE SET NULL, nullable | 创建者 Agent |
| metadata | JSONB | 默认 `{}` |
| created_at | TIMESTAMPTZ | 创建时间 |

约束：`UNIQUE(artifact_id, version)`。索引：`artifact_id`。

> **工件正文与预览：** 工作流执行产生的 `artifact_version` 会把正文上传到 MinIO（`uri` 写入 `s3://...`）；若对象存储不可用，当前链路会直接失败，不再回退伪造存储地址。同时在 `metadata` 中保留 `inline_content`、`source_name`、`stored_in`，供控制台直接预览工件内容。

> **返工版本策略：** 对同一任务下同名且同 `artifact_type` 的工件，返工时不再重复创建新的 `artifact` 记录，而是在原工件上追加新的 `artifact_version`，以保持 PRD / 设计稿等交付物的单工件多版本语义。

### 5.12 review_report

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID, PK | 主键 |
| run_id | UUID, FK → workflow_run ON DELETE SET NULL, nullable | 所属工作流运行，支持按 run 追踪评审 |
| task_id | UUID, FK → task ON DELETE CASCADE | 被评审的任务 |
| reviewer_id | UUID, FK → agent | 评审者 Agent |
| artifact_version_id | UUID, FK → artifact_version ON DELETE SET NULL, nullable | 被评审的工件版本 |
| verdict | TEXT + CHECK | `approved` / `rejected` / `needs_revision` |
| summary | TEXT | 评审总结 |
| findings | JSONB | 发现问题列表，默认 `[]` |
| recommendations | JSONB | 改进建议列表，默认 `[]` |
| metadata | JSONB | 默认 `{}` |
| created_at | TIMESTAMPTZ | 创建时间 |

索引：`run_id`、`task_id`、`reviewer_id`。

> **评审上下文：** 工作流自动生成的 `review_report.metadata` 会记录 `artifact_count` 与 `artifacts[]`，其中包含 `artifact_id`、`version_id`、`name`、`artifact_type`、`version_uri` 与可选 `content_preview`，用于说明评审到底基于哪些交付物完成。

> **任务级评审约束：** Reviewer 在一次任务评审中必须只产出 1 条任务级结论；即使输入中包含多个工件，也必须综合后输出单条 `review_report`，避免对同一任务重复推进状态。

> **评审流转逻辑：** 当 verdict 为 `approved` 时，服务层自动将关联任务推进到 `pending_approval` 并创建 `approval_request`；当 verdict 为 `rejected` 或 `needs_revision` 时，自动将任务状态转为 `revision_required`。

> **阶段门禁：** 默认本地编排引擎下，`approval_request.metadata` 会写入 `run_id` 与 `phase_id`。同一阶段内审批可并行存在；跨阶段推进依赖 `project_phase.sort_order`。当当前阶段审批全部收口后，工作流运行会从 `waiting_approval` 自动恢复到下一阶段。

### 5.14 workflow_run / workflow_step

| 字段 | 类型 | 说明 |
|------|------|------|
| workflow_run.id | UUID, PK | 工作流运行主键 |
| workflow_run.project_id | UUID, FK → project | 所属项目 |
| workflow_run.status | TEXT + CHECK | `pending` / `running` / `waiting_approval` / `waiting_manual_intervention` / `completed` / `failed` / `cancelled` |
| workflow_run.summary | TEXT | 运行摘要；在 `waiting_approval` 或 `waiting_manual_intervention` 时描述当前卡点 |
| workflow_run.error | TEXT | 失败原因；在 `waiting_manual_intervention` 时保留最近一次 LLM 失败信息 |
| workflow_run.started_at / completed_at | TIMESTAMPTZ | 开始/结束时间 |

`waiting_approval` 表示当前阶段审批尚未收口；`waiting_manual_intervention` 表示 PM / Supervisor / Worker / Reviewer 的真实 LLM 调用失败，流程暂停，等待人工修复后通过恢复接口继续执行。

| 字段 | 类型 | 说明 |

| workflow_step.id | UUID, PK | 工作流步骤主键 |
| workflow_step.run_id | UUID, FK → workflow_run | 所属运行 |
| workflow_step.name | TEXT | 步骤名称 |
| workflow_step.agent_role | TEXT | 执行角色 |
| workflow_step.agent_id / task_id / phase_id | UUID, nullable | 关联 Agent、任务、阶段 |
| workflow_step.status | TEXT + CHECK | `pending` / `running` / `completed` / `failed` / `skipped` |
| workflow_step.summary / error | TEXT | 输出摘要 / 错误信息 |
| workflow_step.sort_order | INT | 顺序号 |
| workflow_step.started_at / completed_at / created_at | TIMESTAMPTZ | 时间戳 |

### 5.13 approval_request

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID, PK | 主键 |
| project_id | UUID, FK → project ON DELETE CASCADE | 所属项目 |
| task_id | UUID, FK → task ON DELETE SET NULL, nullable | 关联任务 |
| artifact_id | UUID, FK → artifact ON DELETE SET NULL, nullable | 关联工件 |
| requested_by | UUID, FK → agent | 发起审批的 Agent |
| approver_type | TEXT + CHECK | `user` / `agent` |
| approver_id | TEXT | 审批人标识 |
| title | TEXT, NOT NULL | 审批标题 |
| description | TEXT | 说明 |
| status | TEXT + CHECK | `pending` / `approved` / `rejected` |
| decision_note | TEXT | 审批意见 |
| decided_at | TIMESTAMPTZ, nullable | 决策时间 |
| metadata | JSONB | 默认 `{}` |
| created_at / updated_at | TIMESTAMPTZ | 时间戳 |

索引：`project_id`、`task_id`、`status`。

> **审批依据：** 评审通过自动创建的 `approval_request.metadata` 除 `run_id`、`phase_id` 外，还会写入 `review_id`、`review_summary`、`findings`、`recommendations`、`artifact_count`、`artifacts[]` 等审批上下文，避免审批页面只看到报告编号。

> **审批回写：** approved 时自动将关联任务推进为 `completed`；rejected 时推为 `revision_required`（若任务当前状态允许转换）。

> **恢复告警语义：** 审批决策一旦成功落库，即视为审批成功；若后续读取阶段上下文、解析 `run_id` 或恢复工作流运行失败，接口不会回滚审批结果，而是通过决策结果中的告警字段和 `approval_request.continuation_warning` 审计事件暴露该问题。
