# WP2-01 编排运行态持久化实现总结

## 变更摘要

将编排引擎（Orchestrator Engine）的工作流运行状态从**内存存储**（`RunStore`）迁移到 **PostgreSQL 持久化**。工作流运行（`workflow_run`）和步骤（`workflow_step`）现在持久化到数据库，支持分页查询、按项目/状态过滤，重启后数据不丢失。

## 关键实现说明

### 架构变化
- **之前**：`Engine.store *RunStore`（内存 `map[string]*WorkflowRun`，重启即丢失）
- **之后**：`Engine.workflow *service.WorkflowService`（通过 Repository 层持久化到 PostgreSQL）

### 持久化策略
- 每个步骤的创建、启动、完成/失败都即时写入数据库
- `runCtx` 结构体在执行期间跟踪步骤排序号
- 步骤的 `agent_id`、`task_id`、`phase_id` 在执行过程中设置，完成/失败时一并持久化

### 数据库表设计
- `workflow_run`：UUID 主键，外键 → `project`，状态 CHECK 约束（pending/running/completed/failed）
- `workflow_step`：UUID 主键，外键 → `workflow_run`（CASCADE DELETE），排序字段，可空的 agent/task/phase 关联

### API 变更
- `GET /api/v1/orchestrator/runs` 新增查询参数：`limit`、`offset`、`project_id`、`status`
- `GET /api/v1/orchestrator/runs/{id}` 现在返回含 `steps` 数组的完整运行数据

## 运行方式

```bash
# 1. 启动基础设施
docker-compose up -d postgres

# 2. 执行迁移（包含新的 000007）
docker-compose run --rm migrate

# 3. 启动 API
cd backend && go run ./cmd/api/

# 4. 触发工作流运行
curl -X POST http://localhost:8080/api/v1/orchestrator/runs \
  -H "Content-Type: application/json" \
  -d '{"project_id": "<project-uuid>"}'

# 5. 查询运行列表（支持分页和过滤）
curl "http://localhost:8080/api/v1/orchestrator/runs?limit=10&status=completed"

# 6. 查询单个运行（含步骤）
curl http://localhost:8080/api/v1/orchestrator/runs/<run-uuid>
```

## 已知限制

1. `orchestrator/workflow.go` 中的旧 `RunStore` 和局部类型仍保留（未删除），避免破坏可能的外部引用，后续可清理
2. Repository 层测试需要真实数据库连接，当前仅有模型层单测
3. 工作流运行是同步阻塞执行，长时间运行可能超时（后续可改为异步）
4. 无重试/恢复机制：如果服务在运行中途崩溃，run 状态将停留在 `running`
